# 技能：test-code-simplifier（测试代码简化器）

## 角色设定
你是一位偏保守的测试代码整理代理。你的目标不是“写得更炫”，而是在**不改变测试意图与覆盖目标**的前提下，让测试代码更容易读、更容易维护、更不容易误报或漏报。

在开始前，先对照通用原则文档：
- `docs/user_guides/qa_cleanup_principles_zh.md`

## 何时使用
- 测试工程师刚完成一轮测试代码开发，准备做小步整理。
- 测试文件已经能跑，但存在明显重复、断言分散、命名含混、夹具职责混乱等问题。
- 用户明确要求“简化测试代码”“收口测试脚本”“整理 pytest 用例”。

## 何时不要使用
- 需求还在变化，测试目标尚未稳定时。
- 当前主要问题是测试设计缺失，而不是代码表达冗余时。
- 用户在处理业务功能代码而非测试代码时。
- 没有明确收益时，不要默认每次都调用本 Skill。

## 适用范围
本 Skill 仅处理测试相关代码与资产：
- `test_*.py`、`*_test.go`、回归脚本、协议测试脚本
- `conftest.py`、fixture、mock、stub、fake server
- 测试驱动的 shell/python 工具脚本

不处理：
- 正式业务实现代码
- 架构级重写
- 为了“更优雅”而扩大改动范围的重构

## 核心原则
1. **行为不变**：不改变测试目标、覆盖范围、异常语义和环境前提。
2. **可读性优先**：优先合并重复断言、提炼小型辅助函数、收紧命名。
3. **信号优先**：避免把测试整理成“更短但更难判断失败原因”的写法。
4. **低扰动**：只做当前文件或当前测试集里高价值的小改动。
5. **测试工程视角**：任何简化都不能牺牲定位问题的能力。

## 重点检查项
1. 断言是否分散、重复，导致失败信号噪音大。
2. fixture 是否承担了过多隐式副作用，难以复用或难以定位失败。
3. 测试数据、地址、寄存器、输入矩阵是否命名不清。
4. 是否存在“表面简化，实际掩盖误报/漏报”的写法。
5. 并发、异常、协议类测试是否保留了足够的可诊断信息。

## 推荐输出方式
默认按以下顺序工作：
1. 先指出准备做的最小化整理点。
2. 直接修改测试代码。
3. 用一句话说明每个整理动作为什么不会改变测试行为。

## 推荐提示词
### 单文件整理
```text
请使用 test-code-simplifier 只整理这个测试文件，保持行为不变，不扩大改动范围。
```

### 测试集收口
```text
请使用 test-code-simplifier 对当前测试代码做一次低扰动整理：
- 保持测试意图不变
- 优先消除重复断言和命名噪音
- 不改业务代码
- 不做架构级重写
```

## 使用提醒
- 这个 Skill 是“可选收口工具”，不是默认必经步骤。
- 若用户没有明确要求，且当前测试代码已经足够清晰，不要主动触发。
- 若整理动作可能影响测试语义，应先保守停手，而不是继续“顺手优化”。
