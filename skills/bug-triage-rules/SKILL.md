# Bug 优先级判定规则 (Bug Triage Rules)

为 AI Agent 和测试工程师提供一套**可机械执行的 Bug 严重程度判定规则**，消除人工定级时的主观偏差。

- **输入**：Bug 的现象描述、影响范围、复现条件等事实
- **输出**：**P0 / P1 / P2** 三级严重程度等级
- **复用场景**：`bug-diagnostic-flow` 定级环节、`issue-reporter` 建单时自动填充、独立批量 triage

---

## 一、严重程度等级定义（三级体系）

| 等级 | 名称 | 定义 | 典型信号 |
|:---|:---|:---|:---|
| **P0** | 致命/阻断 | 系统不可用、造成安全/财产风险、或核心业务流程完全中断且无可用 Workaround | 系统宕机、数据丢失不可恢复、安全保护功能失效（过压/过流/过温保护等）、控制指令误动作、主功能完全瘫痪且无备用通道 |
| **P1** | 严重缺陷 | 功能明显异常，影响业务正确性，但存在可接受的 Workaround 或不完全阻断主流程 | 核心流程可走通但需手动规避、数据展示错误可能误导决策、关键功能降级运行、协议合规性偏差（IEC 104 / Modbus 报文异常等）、阻断部分测试项执行 |
| **P2** | 轻微缺陷 | 不影响功能正确性和业务连续性，仅影响易用性、观感或非关键路径 | UI 文案/布局异常、日志格式不规范、非核心功能的体验瑕疵、文档与实际行为不一致 |

### 等级边界速查

| 边界 | 归入上级的信号 | 归入下级的信号 |
|:---|:---|:---|
| **P0 vs P1** | 无任何 Workaround；影响安全或数据完整性；全量用户受影响；保护功能失效 | 有稳定的 Workaround；不涉及安全；影响范围可控 |
| **P1 vs P2** | 影响业务数据正确性；用户需要额外操作才能完成正常流程；协议层面偏差 | 纯展示/体验层面；不影响任何业务逻辑和数据 |

> **并发/资源竞争类缺陷特别说明**：此类问题常表现为"偶发"，但本质是设计层面的遗漏（如总线仲裁、锁机制缺失），修复成本高且难以临时规避。定级时**不应因"偶发"而降级**，应重点评估其对业务连续性的实际影响。

## 二、判定流程（四步决策树）

依次评估以下四个问题，命中即停止：

```
问 1：是否涉及安全风险（含保护功能失效）或 数据不可恢复？
├── 是 → P0
└── 否 →
    问 2：是否阻断核心业务流程 且 无可用 Workaround？
    ├── 是 → P0
    └── 否 →
        问 3：是否影响功能正确性 或 业务数据准确性？
        ├── 是 → P1（有稳定 Workaround 的阻断也归入此级）
        └── 否 → P2
```

### 判定因子详解

#### 因子 1：安全性影响 (Safety Impact)
> 该缺陷是否可能导致人身安全风险或设备物理损坏？保护功能（过压/过流/过温等）是否失效？

- **是** → 直接判定 **P0**，无需继续评估
- **否** → 继续下一因子

#### 因子 2：数据完整性 (Data Integrity)
> 该缺陷是否导致数据丢失、数据污染、或不可恢复的状态异常？

- **是** → 判定 **P0**
- **否** → 继续下一因子

#### 因子 3：核心业务阻断度 (Business Blocking)
> 该缺陷是否阻断了主业务流程？是否存在可用的 Workaround？

| 阻断主流程？ | 有 Workaround？ | 判定 |
|:---|:---|:---|
| 是 | 否 | **P0** |
| 是 | 是（但代价高/不稳定） | **P1** |
| 是 | 是（简单且稳定） | **P1** |
| 否 | - | 继续下一因子 |

#### 因子 4：功能正确性影响 (Functional Correctness)
> 该缺陷是否影响了业务数据的正确性、功能的预期行为、或协议合规性？

- **是** → 判定 **P1**
- **否** → 判定 **P2**

---

## 三、输出格式

定级完成后，输出以下结构化结论：

```markdown
## 严重程度定级

- **等级**：P0 / P1 / P2
- **命中因子**：（如：因子 1 - 安全性影响）
- **判定依据**：（一句话说明为什么命中该因子）
```
