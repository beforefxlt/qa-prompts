# 测试驱动开发辅助 (test-driven-development)

本 Skill 带有明显的通用工程方法论来源，当前仓库**暂不把它作为推荐主线能力**。  
它保留为**按需启用的工程纪律辅助 Skill**，适用于用户明确要求按 TDD 方式推进开发或修复问题的场景。

## 何时使用

- 用户明确要求使用 TDD
- 当前任务是功能实现、缺陷修复、行为调整
- 仓库已有可执行测试体系，能够快速验证失败与通过

## 何时不要使用

- 用户要的是测试策略、测试计划、测试用例、测试报告
- 当前只是做文档梳理、提示词治理或资产复核
- 项目没有可执行测试入口，或当前任务不值得引入完整 TDD 循环

## 最小方法

只保留最核心的 `Red -> Green -> Refactor`：

1. **Red**  
   先写一个能表达目标行为的最小失败测试，并确认它确实因目标能力缺失而失败。
2. **Green**  
   只写让该测试通过的最小实现，不顺手扩展需求。
3. **Refactor**  
   在测试保持通过的前提下整理命名、重复逻辑和局部结构。

## 执行要点

- 一个测试只表达一个行为
- 先看测试失败，再开始实现
- 先跑最小相关测试，再决定是否扩到更大范围
- 若修的是 bug，优先先补可复现该 bug 的失败测试

## 输出要求

若调用本 Skill，输出应至少说明：

- 本轮目标行为
- 新增或修改了哪个测试
- 失败是如何被确认的
- 通过是如何被确认的
- 是否还有未覆盖的边界风险

## 使用提醒

- 它是工程纪律辅助，不是 QA 主链技能
- 不要把它当成“所有任务都必须自动触发”的默认规则
- 若用户没有明确要求，且当前任务以文档/测试资产治理为主，不要主动启用
