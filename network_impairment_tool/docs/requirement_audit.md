# 需求可测试性与歧义审查（Shift-Left Testing）- 网络损伤模拟工具

**版本**: v1.0
**关联功能**: network_impairment_tool
**审计人**: Antigravity (Test Architect)

| 需求点 | 潜在风险/歧义 | 建议优化方案 | 风险等级 |
| :--- | :--- | :--- | :--- |
| **透明桥接架构** | 桥接配置失败可能导致主机失联（SSH中断） | 增加“管理网口白名单”说明，或明确建议在独立网口（非管理口）上操作 | **高** |
| **随机波动参数** | “随机”缺乏边界定义，可能出现极端组合导致链路彻底断开 | 在脚本中设置硬性上限（如丢包率不超过 20%，延迟不超过 2000ms） | **中** |
| **延迟与抖动** | 抖动值（Jitter）大于延迟值（Delay）时，会导致数据包重排或丢弃 | 脚本逻辑应校验 `jitter < delay`，防止内核 netem 报错或产生非预期的严重丢包 | **高** |
| **一键清理** | 强制终止（如 `kill -9`）可能导致 `tc` 规则残留在内核中 | 文档中补充手动清理命令 `tc qdisc del dev <iface> root`，作为兜底方案 | **低** |
| **性能损耗** | 损伤主机本身在高吞吐量下的 CPU 瓶颈未定义 | 明确支持的最大带宽（如 100Mbps 或 1Gbps），防止损伤工具自身成为性能瓶颈 | **中** |
| **多用户冲突** | 若多个进程同时操作同一个网口的 tc 规则会冲突 | 增加文件锁或 PID 检查，防止重复启动 | **低** |
| **可测量性** | 如何验证“损伤”确实符合预期？ | 建议集成一个简易的监控逻辑，或在 README 中提供验证方法（如对端 ping 测试） | **中** |
