# 全景测试策略设计 (Test Strategy Matrix) - 随机网络损伤工具

**版本**: v1.0
**工具对象**: `network_impairment_tool/random_impairment.py`
**设计维度**: P1-P5 全矩阵覆盖

---

## 策略大纲

### P1: 基础路径 (Happy Path) - 功能闭环验证
- **用例 1.1**: 单一损伤注入验证。分别测试只有 `delay`、只有 `loss` 的场景，检查 `tc qdisc show` 输出是否精准。
- **用例 1.2**: 组合损伤注入验证。测试 `delay + jitter + loss` 同时生效的场景。
- **用例 1.3**: 自动恢复验证。运行脚本后按 `Ctrl+C` 正常退出，验证 `tc` 规则是否被彻底清除。

### P2: 数据与业务异常 (Domain Failure) - 参数健壮性
- **用例 2.1**: 非法接口名称。传入一个不存在的网口名称（如 `eth99`），系统应友好提示错误而非崩溃。
- **用例 2.2**: 重复启动冲突。在同一网口上同时运行两个脚本实例，验证冲突处理逻辑。
- **用例 2.3**: 权限不足验证。非 `root` 用户运行脚本，验证是否准确拦截并提示 `sudo`。

### P3: 环境与基建异常 (Infrastructure Error) - 极致健壮性
- **用例 3.1**: 缺少依赖工具。在未安装 `iproute2` 的纯净容器/环境中运行，验证错误捕获。
- **用例 3.2**: 接口状态突变。脚本运行过程中，手动 `ip link set br0 down`，验证脚本是否能感知并记录日志。
- **用例 3.3**: 强制终止残留测试。模拟 `kill -9` 杀死进程，验证规则残留情况，确认 README 中的“手动清理指令”有效。

### P4: 边界条件 (Boundary & Edge cases) - 极限压力
- **用例 4.1**: 抖动临界值。测试 `jitter = delay - 1` 的组合，验证修正逻辑是否生效。
- **用例 4.2**: 损伤值极限。测试 `loss = 100%`（全断路）和 `loss = 0%` 的边界表现。
- **用例 4.3**: 时间间隔极限。测试 `interval = 1s` 的极高频切换，观察系统资源占用情况。

### P5: 交付与最终使用视角 (Acceptance) - 落地体验
- **用例 5.1**: 透明桥接一键部署。按照 README 步骤配置 `br0`，验证数据链路联通性（ping 对端）。
- **用例 5.2**: 诊断体验验证。观察脚本输出的日志是否清晰（是否包含了具体的延迟/丢包数值）。
- **用例 5.3**: SSH 会话稳定性。在非管理口进行损伤时，确认 SSH 窗口无卡顿，无掉线。
