# 代码回归测试流水线 (Regression Test Workflow)

> **触发条件**：当用户要求对已有代码进行审查、修复 Bug、或做回归测试时，启动本流程。
> **核心原则**：先审查发现 Bug，再写 UT 修复（先红后绿），再跑全链路，最后对齐文档。
> **设计动机**：`test-lifecycle.md` 只覆盖"从 0 到 1 新建项目"的测试流程，缺少"已有代码回归"的场景。本工作流填补这一空白。

---

## Step 0: 变更范围确认

1. **扫描未提交变更**：运行 `git diff` 和 `git status --short`。
2. **识别变更类型**：
   - 新功能代码 → 转入 `test-lifecycle.md`
   - Bug 修复代码 → 继续本流程
   - 纯重构代码 → 跳到 Step 4（全链路回归）
3. **向用户汇报变更摘要**，确认本轮回归范围。

### Step 0 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 变更范围已确认 | 用户明确了本轮要审查/回归的文件和模块 |

---

## Step 1: 代码审查（强制触发）

> **本步骤是强制的，不依赖用户要求。所有代码变更在进入测试前必须经过审查。**

1. **全量 diff 审查**：运行 `git diff` 查看所有未提交变更。
2. **逐个文件阅读**：不能只看 diff，必须阅读每个变更文件的完整内容。
3. **审查清单**（来自真实 Bug 模式库）：

### L1: 逻辑错误

| 模式 | 检查方式 | 来源 Bug |
|------|---------|---------|
| 三元表达式两分支相同 | `x ? a : a` 永远返回 `a` | BUG-044 |
| 时间序列直接索引取"最新" | `series[0]` = 最旧值，不是最新值 | BUG-046 |
| 未排序取最新/次新 | `series.filter()` 后取 `length-1`，不一定是最新 | BUG-050 |
| 空状态判断逻辑错误 | 条件组合遗漏（如 `!isChild \|\| !visionData`） | BUG-047 |

### L2: 契约一致性

| 模式 | 检查方式 | 来源 Bug |
|------|---------|---------|
| API 路径缺少参数 | 函数接收了参数但 URL 中未使用 | BUG-045 |
| 响应字段与 schema 不一致 | 测试断言字段不存在于实际响应中 | test_full_pipeline.py 3 轮迭代 |
| 枚举值/类型不匹配 | 前端期望的字段后端未返回 | BUG-034/036/037/038 |

### L3: 环境相关

| 模式 | 检查方式 | 来源 Bug |
|------|---------|---------|
| 硬编码 localhost/固定端口 | `constants/` 中出现 `localhost` | BUG-049 |
| 硬编码绝对路径 | 路径不跨平台 | NP-05 |

### L4: 代码质量

| 模式 | 检查方式 |
|------|---------|
| 未使用的 import/变量 | ESLint `no-unused-vars` |
| 重复代码 | 组件内定义了函数但未使用 |
| 规格遗漏 | `UI_SPEC.md` 定义了 N 个页面但代码只有 M 个 |

4. **记录发现的问题**：写入 `/tmp/code_review_findings.md`。

### Step 1 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 审查文件已落盘 | `/tmp/code_review_findings.md` 存在且非空 |
| 每个问题有来源标注 | 标注了对应的 Bug 编号或审查清单类别 |
| 用户确认 | 向用户展示问题列表，用户回复"进入 Step 2" |

---

## Step 2: Bug 修复（TDD 纪律）

> **对 Step 1 发现的每个问题，严格执行：先写失败的 UT → 修复 → 验证通过。**

对每个发现的问题执行以下循环：

1. **写失败的 UT**：
   - 在对应的测试文件中添加回归用例
   - 运行测试确认失败（红）
   - 如果测试意外通过，说明问题不存在或已修复，跳过该问题

2. **修复代码**：
   - 优先提取纯函数到 `utils/`（提高可测性）
   - 修复组件/服务中的 bug
   - 不引入新的 mock/stub 逻辑到生产代码

3. **验证通过**：
   - 运行该测试文件确认通过（绿）
   - 运行全量测试确认无回退

### Step 2 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 每个 Bug 有 UT 覆盖 | 每个修复的问题至少有一个回归用例 |
| 全量测试通过 | `npm test` / `pytest` 全部通过 |
| 无生产代码中的测试逻辑 | `check_no_test_code.py` 输出 PASS |

---

## Step 3: 全链路集成测试

> **验证修复后的代码在完整调用链中正常工作。**

### 3.1 后端集成测试

- 使用 `httpx.AsyncClient` + in-memory SQLite
- 串联完整 API 调用链（创建→上传→OCR→审核→查询）
- mock 外部依赖（OCR 服务），但走真实代码路径
- **使用真实测试图片**（`backend/tests/01.jpg`），禁止使用 `b"dummy-content"` 假数据

### 3.2 移动端全链路测试

- 使用 mock fetch 串联服务层调用
- 覆盖正常链路 + 异常链路 + 幂等 + 弱网 + 数据污染
- 验证各服务之间的数据传递正确性

### 3.3 前端 E2E 测试（如环境可用）

- 使用 Playwright 驱动浏览器
- 测试开头必须包含环境自检（`fetch('/health')`，不通则 `test.skip()`）

### Step 3 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 后端集成测试通过 | 全链路 API 调用链无阻断 |
| 移动端全链路测试通过 | 正常 + 异常 + 幂等 + 弱网场景全部覆盖 |
| E2E 测试通过或跳过 | 环境可用时通过，不可用时 skip（不报错） |

---

## Step 4: 失败路径测试

> **验证系统在异常场景下的容错能力。**

### 4.1 弱网场景

- 超时后重试成功
- 断线重连后操作恢复
- 慢响应后不阻塞 UI

### 4.2 幂等性验证

- 重复上传同一文件返回 `duplicate`，不创建新记录
- 重复提交 OCR 不产生重复数据
- 重复审核同一任务不重复创建数据

### 4.3 服务端错误恢复

- 500/502/503 后重试成功
- 连续多次失败后放弃重试

### 4.4 数据污染防护

- 上传成功但 OCR 失败 → Dashboard 显示空数据（不产生脏数据）
- 审核通过但趋势查询失败 → 数据已持久化（不丢失）

### Step 4 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 弱网场景覆盖 | 超时/断线/慢响应均有测试 |
| 幂等性验证通过 | 重复操作不产生副作用 |
| 数据污染防护通过 | 部分失败不产生脏数据 |

---

## Step 5: 文档对齐

> **确保代码变更与文档保持同步。**

1. **BUG_LOG.md**：新增的 Bug 修复记录已写入
2. **AGENTS.md**：新的负向提示词已添加（如有）
3. **工作流文档**：如有新的测试模式/审查模式，已更新对应工作流
4. **STATUS.md**：项目状态已同步
5. **运行 `check_docs_alignment.py`**：自动化检查

### Step 5 门禁

| 检查项 | 通过条件 |
|:---|:---|
| BUG_LOG.md 已更新 | 新增 Bug 修复记录存在 |
| 文档对齐检查通过 | `check_docs_alignment.py` 无阻断项 |
| 生产代码检查通过 | `check_no_test_code.py` 输出 PASS |

---

## Step 6: 最终复核

1. **运行全量测试**：`npm test` + `pytest`，确认 100% 通过。
2. **检查测试覆盖率**：确认覆盖率不低于回归前水平。
3. **产出回归报告**：写入 `/tmp/regression_report.md`，包含：
   - 审查发现的问题数量及严重程度
   - 修复的 Bug 列表及 UT 覆盖情况
   - 全链路测试结果
   - 失败路径测试结果
   - 文档对齐状态

### Step 6 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 全量测试通过 | 0 failed |
| 覆盖率不降低 | 语句覆盖率 >= 回归前水平 |
| 回归报告已落盘 | `/tmp/regression_report.md` 存在且非空 |
| 用户确认 | 用户确认可以 commit |
