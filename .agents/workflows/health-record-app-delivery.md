# 家庭检查单管理应用 AI 交付流水线

> **触发条件**：当用户准备从 0 到 1 开发"家庭检查单管理应用"或同类健康数据管理产品时，启动本流程。
> **核心原则**：先规格冻结，再测试驱动，再按明确边界拆 Subagent 并行开发。禁止跳过 UI 规格、OCR 样本与测试门禁直接写实现。

## Step 0: 强制资产嗅探与基线锚定 (Pre-flight Check)

> **红线规则**：本步骤为所有后续步骤的绝对前提。跳过此步的任何产出物一律视为无效。

1. **扫描项目正式文档目录**：执行 `list_dir` 对 `docs/specs/` 进行全量扫描。
2. **精读已有设计文档**：对扫描到的每一份 `.md` 文件（如 `PRD.md`, `ARCHITECTURE.md`, `DATABASE_SCHEMA.md`, `API_CONTRACT.md`, `UI_SPEC.md` 等）执行 `view_file` 完整阅读。
3. **锚定事实源**：将已有文档中明确的技术选型（如后端框架、数据库类型、中间件）、数据模型（表结构与字段）、安全约束等提取为**不可推翻的基线约束**。
4. **输出基线锚清单**：在 `/tmp/baseline_anchor.md` 中列出从正式文档中提取的所有硬约束（如："后端: FastAPI", "数据库: PostgreSQL", "存储: MinIO"）。
5. **后续步骤的所有产出必须与此基线保持一致，任何偏离必须经用户显式批准。**

## Step 1: 产品与风险收敛

1. **基于 Step 0 锚定的基线**，识别 PRD 中尚未覆盖的需求真空（只问增量问题，已有文档覆盖的内容严禁重复提问）。
2. 使用 `requirement-reviewer` 审计剩余的隐私风险、P3/P5 缺口。
3. 将阶段结论写入 `/tmp/health_record_requirement_audit.md`，并在文件头部声明："本文件基于 docs/specs/ 下的正式文档生成，已对齐基线约束"。
4. 用户确认需求范围后再进入下一步。

## Step 2: UI 规格冻结

1. 输出信息架构、页面清单、关键任务流、低保真线框要求。
2. 重点固化上传、OCR 审核、趋势图、报警红区四类页面。
3. 将结果写入 `/tmp/health_record_ui_spec.md`。
4. 用户确认页面结构后再进入架构设计。

## Step 3: 技术架构与数据模型设计

1. 输出系统模块图、数据模型、OCR 链路、规则引擎边界、部署结构。
2. 明确原始抽取结果与正式 observation 的分层关系。
3. 将结果写入 `/tmp/health_record_architecture.md`。
4. 用户确认架构后再进入测试设计。

## Step 4: 测试策略冻结

1. 使用 `test-strategy-planner` 输出 P1-P5 测试矩阵。
2. 强制补齐 OCR Golden Set、规则引擎、E2E、隐私与容灾测试。
3. 将结果写入 `/tmp/health_record_test_strategy.md`。
4. 用户确认测试门禁后再进入实施计划。

## Step 5: 实施计划拆解

1. **输入来源强制校验**：实施计划的输入规格**必须且只能**引用 `docs/specs/` 下的正式文档（如 `ARCHITECTURE.md`, `DATABASE_SCHEMA.md`）。严禁以 `/tmp/` 目录下的临时草案作为技术选型或数据模型的依据。
2. 使用 `writing-plans` 思路拆成可执行任务。
3. 每个任务必须写清：目标、输入规格（标注引用的正式文档路径）、输出文件、测试方式、禁止修改范围。
4. 任务粒度以 2 到 8 小时为宜，避免"大而全任务包"。
5. 计划文件进入正式仓库路径。

## Step 6: Subagent 并行开发

> **平台能力适配声明**：当前 `browser_subagent` 仅具备浏览器自动化能力（截图、点击、DOM 读取），**不具备**文件写入和终端命令执行能力。因此：
> - `browser_subagent` 仅用于 **UI 验证、E2E 测试、截图对比**等浏览器域任务。
> - 后端编码类任务（API / OCR / Data）由主 Agent 在主线程中**按任务包顺序串行执行**，每完成一个任务包后执行测试验证，再进入下一个。
> - **降级透明原则**：若任何 subagent 调用失败或能力不足，Agent **必须立即向用户报告**，严禁静默降级为其他执行模式。

1. 使用 `subagent-driven-development` 执行计划。
2. 按写域拆分 `UI / API / OCR / Data / QA`。
3. 每个任务包必须先写失败测试，再实现最小代码。
4. 每个任务完成后必须依次执行：
   - 规格一致性复核
   - 代码质量复核

## Step 7: 集成与回归

1. 合并实现后跑单元测试、API 合约测试、集成测试。
2. 跑 OCR Golden Set 回归。
3. 跑前端 E2E 与关键页面截图检查。
4. 若隐私脱敏、单位校验或审核状态流失败，直接阻断交付。

## Step 8: 核验与收口

1. 使用 `verify-requirements` 逐条核验规格是否落地。
2. 使用 `reviewer-agent` 做最终一致性与文档同步复核。
3. 若文档、测试、架构说明未同步，不允许进入提交阶段。
