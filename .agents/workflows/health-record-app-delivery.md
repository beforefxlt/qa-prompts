# 家庭检查单管理应用 AI 交付流水线

> **触发条件**：当用户准备从 0 到 1 开发"家庭检查单管理应用"或同类健康数据管理产品时，启动本流程。
> **核心原则**：先规格冻结，再交互确认，再测试驱动，再按明确边界拆 Subagent 并行开发。禁止跳过 UI 交互原型与测试门禁直接写实现。
> **v2.2.0 变更说明**：新增 Step 6.5（代码审查，强制触发）、Step 7.1 增强（全链路集成测试 + 失败路径测试 + 真实测试图片）。

---

## Step 0: 强制资产嗅探与路径分流 (Pre-flight Check)

> **红线规则**：本步骤为所有后续步骤的绝对前提。跳过此步的任何产出物一律视为无效。

1. **扫描项目正式文档目录**：执行 `list_dir` 对 `docs/specs/` 进行全量扫描。
2. **精读已有设计文档**：对扫描到的每一份 `.md` 文件执行 `view_file` 完整阅读。
3. **锚定事实源**：将已有文档中明确的技术选型、数据模型、安全约束等提取为**不可推翻的基线约束**。
4. **成熟度评估与路径分流**：根据扫描结果，按以下矩阵判断项目成熟度，决定跳过哪些步骤：

| 已有文档 | 覆盖的步骤 | 若存在则 |
|:---|:---|:---|
| `PRD.md` | Step 1 (需求收敛) | **跳过 Step 1**，只做增量差距分析 |
| `UI_SPEC.md` | Step 2a (信息架构) | **跳过 Step 2a** |
| `UI_SPEC.md` 中含页面导航流程图或存在独立交互原型文件 | Step 2b (交互原型) | **跳过 Step 2b** |
| `ARCHITECTURE.md` + `DATABASE_SCHEMA.md` | Step 3 (架构设计) | **跳过 Step 3** |
| `TEST_STRATEGY.md` | Step 4 (测试策略) | **跳过 Step 4** |

5. **输出基线锚清单与跳转决策**：在 `/tmp/baseline_anchor.md` 中列出：
   - 所有硬约束（如："后端: FastAPI", "数据库: PostgreSQL"）
   - 跳转决策（如："Step 1-4 已由正式文档覆盖，直接进入 Step 5"）
6. **向用户汇报评估结果**，经用户确认后按决策路径执行。

### Step 0 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 基线锚清单已落盘 | `/tmp/baseline_anchor.md` 文件存在且非空 |
| 用户已确认跳转决策 | 用户明确回复"确认"或指出修正项后再次确认 |

## Step 1: 产品与风险收敛

1. **基于 Step 0 锚定的基线**，识别 PRD 中尚未覆盖的需求真空（只问增量问题，已有文档覆盖的内容严禁重复提问）。
2. 使用 `requirement-reviewer` 审计剩余的隐私风险、P3/P5 缺口。
3. 将阶段结论写入 `/tmp/health_record_requirement_audit.md`，并在文件头部声明："本文件基于 docs/specs/ 下的正式文档生成，已对齐基线约束"。
4. 用户确认需求范围后再进入下一步。

### Step 1 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 审计文件已落盘 | `/tmp/health_record_requirement_audit.md` 存在且非空 |
| 核心业务流程数量已确认 | 审计文件中明确列出了 PRD 定义的所有业务流程编号 |
| 页面需求已提取 | 审计文件中包含"本产品需要 N 个页面"的明确声明 |
| 用户确认 | 向用户展示风险摘要表，用户回复"进入 Step 2a" |

## Step 2a: UI 信息架构与页面规格

1. 输出信息架构、页面清单、每个页面的展示内容与交互逻辑。
2. 重点固化上传、OCR 审核、趋势图、报警红区四类页面。
3. 将结果写入 `/tmp/health_record_ui_spec.md`。

### Step 2a 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 页面数量一致性 | UI 规格中的页面数量 >= PRD 业务流程所隐含的最少页面数 |
| 交互逻辑完整性 | 每个页面都同时包含"展示内容"和"交互逻辑"两个段落 |
| 状态设计覆盖 | 包含空状态、处理中、异常状态、成功状态四类设计 |
| 文件已落盘 | `/tmp/health_record_ui_spec.md` 存在且非空 |
| 用户确认 | 向用户展示完整页面清单表格，用户回复"进入 Step 2b" |

> **否决条件**：如果 PRD 中描述了 N 条用户任务链路，但 UI 规格中的页面无法覆盖其中任何一条链路的完整闭环，必须打回补齐。

## Step 2b: UI 交互原型与导航流确认

> **本步骤是 v2.0.0 的核心新增。v1.x 的教训表明：纯文字 UI 规格无法有效约束 Agent 的实现范围，导致页面选择性丢失。**

1. **产出页面导航流程图**（必选）：
   - 使用 Mermaid 绘制完整的页面导航流程图
   - 节点 = Step 2a 中定义的每一个页面（一个都不能少）
   - 边 = 用户操作触发的页面跳转（标注触发动作，如"点击成员卡片"）
   - 将流程图嵌入 UI_SPEC.md 末尾或作为独立文件 `UI_NAVIGATION_FLOW.md`

2. **产出可视化原型**（按项目规模二选一）：
   - **方案 A: HTML 低保真原型**（推荐，适合 5+ 页面的项目）
     - 每个页面一个 HTML 文件，只需灰色块 + 文字标注 + 可点击的导航链接
     - 存放于 `/tmp/ui_prototype/` 目录
     - 用浏览器打开后可以点击跳转，体验完整导航流
   - **方案 B: 逐页布局草图描述**（适合 3 页以下的简单项目）
     - 每个页面用 ASCII 或结构化文字描述布局分区
     - 标注每个分区的内容和操作

3. **向用户展示原型**，逐页走查确认。

### Step 2b 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 导航流程图存在 | UI_SPEC.md 末尾包含 Mermaid 流程图，或 `UI_NAVIGATION_FLOW.md` 存在 |
| 全页面覆盖 | 流程图中的节点数量 = Step 2a 页面清单中的页面数量（一个不少） |
| 无孤岛页面 | 每个节点至少有一条入边和一条出边（首页除外，首页只需出边） |
| 用户操作标注 | 每条边都标注了触发动作（如"点击 XX 按钮"） |
| 可视化原型存在 | `/tmp/ui_prototype/` 目录非空，或逐页布局描述已包含在文档中 |
| 用户逐页确认 | 用户回复"进入 Step 3" |

> **否决条件**：如果流程图中的页面数量少于 Step 2a 定义的页面数量，**必须停下来补齐**，严禁以"后续迭代再加"为由跳过。

### Step 4 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 测试策略文件已落盘 | `/tmp/health_record_test_strategy.md` 存在且非空 |
| P1-P5 全层覆盖 | 五个优先级层次均有至少一个测试场景 |
| OCR Golden Set 定义 | 至少定义了一组真实样本的预期输出 |
| 用户确认 | 用户回复"进入 Step 5" |

## Step 5: 实施计划拆解

1. **输入来源强制校验**：实施计划的输入规格**必须且只能**引用 `docs/specs/` 下的正式文档（如 `ARCHITECTURE.md`, `DATABASE_SCHEMA.md`）。严禁以 `/tmp/` 目录下的临时草案作为技术选型或数据模型的依据。
2. 使用 `writing-plans` 思路拆成可执行任务。
3. 每个任务必须写清：目标、输入规格（标注引用的正式文档路径）、输出文件、测试方式、禁止修改范围。
4. 任务粒度以 2 到 8 小时为宜，避免"大而全任务包"。
5. **前端任务必须引用 Step 2b 的交互原型作为实现参照**，而非仅依赖文字规格。
6. 计划文件进入正式仓库路径。

### Step 5 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 实施计划已落盘 | 正式仓库路径下存在实施计划文件 |
| 任务覆盖全部页面 | 实施计划中的前端任务合计覆盖了 Step 2a 定义的全部页面 |
| 每个任务有测试方式 | 每个任务块中都包含"测试方式"段落 |
| 无 /tmp 引用 | 实施计划中不引用 `/tmp/` 作为技术选型依据 |
| 用户确认 | 用户回复"开始执行" |

## Step 6: Subagent 并行开发

> **平台能力适配声明**：当前 `browser_subagent` 仅具备浏览器自动化能力（截图、点击、DOM 读取），**不具备**文件写入和终端命令执行能力。因此：
> - `browser_subagent` 仅用于 **UI 验证、E2E 测试、截图对比**等浏览器域任务。
> - 后端编码类任务（API / OCR / Data）由主 Agent 在主线程中**按任务包顺序串行执行**，每完成一个任务包后执行测试验证，再进入下一个。
> - **降级透明原则**：若任何 subagent 调用失败或能力不足，Agent **必须立即向用户报告**，严禁静默降级为其他执行模式。

1. 使用 `subagent-driven-development` 执行计划。
2. 按写域拆分 `UI / API / OCR / Data / QA`。
3. 每个任务包必须先写失败测试，再实现最小代码。
4. 每个任务完成后必须依次执行：
   - 规格一致性复核
   - 代码质量复核

### Step 6 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 每个任务的测试通过 | 任务包中定义的测试全部 pass |
| 无静默降级 | 执行日志中无未报告的 subagent 失败记录 |
| 进入 Step 6.5 | 所有实施计划中的任务包均标记为完成 |

## Step 6.5: 代码审查 (v2.2.0 新增)

> **本步骤是强制触发的，不依赖用户要求。所有代码变更在进入测试阶段前必须经过审查。**
> **执行时机**：开发完成后、写测试前。先审查发现 Bug，再写 UT 修复。

1. **全量 diff 审查**：运行 `git diff` 查看所有未提交变更。
2. **逐个文件阅读**：不能只看 diff，必须阅读每个变更文件的完整内容。
3. **重点检查**：
   - 三元表达式两分支是否相同（`x ? a : a`）
   - 时间序列数据是否排序后取最新值（禁止 `series[0]` 当最新值）
   - API 路径是否与契约一致（参数是否都在 URL 中使用）
   - 常量是否硬编码（`localhost`、固定端口）
   - 未使用的 import/变量、重复代码
4. **发现问题后**：
   - 先写失败的 UT 确认 bug 存在
   - 修复代码
   - 运行 UT 验证通过
   - 更新 BUG_LOG.md

### Step 6.5 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 代码审查已完成 | 所有变更文件已审查 |
| 发现的 Bug 已修复 | 所有发现的问题已修复并有 UT 覆盖 |
| 全量测试通过 | `npm test` / `pytest` 全部通过 |
| 进入 Step 7 | 代码审查完成且无遗留问题 |

## Step 7: 集成、回归与页面完整性检查

### 7.1 常规集成回归

1. 合并实现后跑单元测试、API 合约测试、集成测试。
2. **全链路集成测试**（v2.2.0 新增）：
   - 后端：使用 `httpx.AsyncClient` + in-memory SQLite，串联完整 API 调用链（创建→上传→OCR→审核→查询），使用真实测试图片（`backend/tests/01.jpg`）。
   - 前端 E2E：Playwright 驱动浏览器，通过真实后端完成端到端验证。
   - 移动端：mock fetch 串联服务层调用，覆盖正常链路 + 异常链路 + 幂等 + 弱网 + 数据污染。
3. **失败路径测试**（v2.2.0 新增）：
   - 弱网（超时、断线重连、慢响应）
   - 幂等性（重复操作不产生副作用）
   - 服务端错误（500/502/503 重试策略）
   - 数据污染防护（部分失败不产生脏数据）
4. 跑 OCR Golden Set 回归。
5. 跑前端 E2E 与关键页面截图检查。
6. 若隐私脱敏、单位校验或审核状态流失败，直接阻断交付。

### 7.2 前端页面完整性检查（v2.0.0 新增）

> **本检查是防止"规格定义了但实现丢失"的最后一道防线。**

1. **提取 UI 规格页面清单**：读取 `UI_SPEC.md`，枚举所有定义的页面。
2. **扫描前端路由文件**：扫描 `frontend/src/app/` 下的实际目录结构和 `page.tsx` 文件。
3. **产出差异对照表**：

| 类型 | 含义 | 处置 |
|:---|:---|:---|
| 规格有、代码无 | 页面未实现 | **阻断** -- 必须实现或从 UI_SPEC.md 中正式移除（需用户批准） |
| 代码有、规格无 | 未定义的页面 | **警告** -- 需要补充到 UI_SPEC.md 或说明为临时页面 |

4. **差异表必须为空（或全部标记为已知豁免）才能进入 Step 8。**

### 7.3 契约同步与文档对齐检查（v2.1.0 新增）

> **本检查是防止"契约断裂"和"文档不同步"的关键门禁。**

1. **契约同步检查**：使用 `@contract-first` skill 验证
   - 检查 `app/schemas/` 变更是否已同步到 `API_CONTRACT.md`
   - 检查前端 `METRIC_OPTIONS` 是否与后端 `validate_sanity_range` 对齐
   - 检查新增 API 是否已添加到契约文档

2. **文档对齐检查**：运行 `check_docs_alignment.py`
   - 检查代码变更是否同步更新相关文档
   - 检查新增功能是否同步到 BUG_LOG.md（如有 Bug 修复）

3. **生产代码检查**：运行 `check_no_test_code.py`
   - 检查 `app/` 目录中是否包含测试逻辑（mock/stub/fixture）
   - 检查是否有 `# TODO: remove before production` 等标记

### Step 7 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 单元测试全通过 | `pytest` 输出 0 failed |
| 前端类型检查通过 | `tsc --noEmit` 输出 0 errors |
| 页面完整性差异表为空 | 无"规格有代码无"的阻断项 |
| Golden Set 回归通过 | 关键字段准确率达标 |
| 契约同步检查通过 | `@contract-first` 验证无阻断项 |
| 文档对齐检查通过 | `check_docs_alignment.py` 无阻断项 |
| 生产代码检查通过 | `check_no_test_code.py` 输出 PASS |
| 用户确认 | 用户回复"进入 Step 8" |

## Step 8: 核验与收口

### 8.1 需求核验

1. 使用 `verify-requirements` 逐条核验规格是否落地。

### 8.2 修复验证（v2.1.0 新增）

> **本检查是防止"声称修复但实际未验证"的关键门禁。**

1. **后端验证**：运行 `pytest`
   - 必须全部通过，不允许有失败的测试用例

2. **前端验证**：运行 `npm run build`
   - TypeScript 编译必须通过
   - Next.js 生产构建必须成功

3. **快速验证**：运行 `verify_fix.py`（如存在）
   - 综合验证后端 API、前端构建、数据库状态

### 8.3 最终复核

1. 使用 `reviewer-agent` 做最终一致性与文档同步复核。
2. 若文档、测试、架构说明未同步，不允许进入提交阶段。

### Step 8 门禁

| 检查项 | 通过条件 |
|:---|:---|
| 需求核验通过 | `verify-requirements` 输出中无阻断项 |
| 后端测试通过 | `pytest` 输出 0 failed |
| 前端构建通过 | `npm run build` 输出 0 errors |
| 文档同步 | 所有规格文档的"最后更新"日期 >= 最近一次代码提交日期 |
| reviewer-agent 无阻断项 | 复核结论中"必须修正"类别为空 |
| 用户确认 | 用户确认可以进入提交阶段 |

## 增量开发入口 (Incremental Development Entry)

> **适用场景**：项目已完成首次交付 (v1.x+)，需要增量补齐页面、修改交互或接入新能力时。
> **设计动机**：v1.1.0 工作流只覆盖"从0到1"，导致迭代开发时 Agent 直接跳过所有门禁开始改代码。

### 入口流程

1. **读取当前状态**：执行 `view_file` 读取 `CURRENT_STATUS.md`，确认当前版本和已实现范围。

2. **页面差距分析**：
   - 读取 `UI_SPEC.md` 的完整页面清单
   - 扫描 `frontend/src/app/` 下的实际路由文件
   - 产出差距清单，标注哪些页面已实现、哪些缺失

3. **用户确认本轮迭代范围**：
   - 向用户展示差距清单
   - 用户选择本轮要实现的页面子集
   - 用户可调整或新增 UI_SPEC.md 中的页面定义

4. **对本轮新页面执行 Step 2b**：
   - 更新页面导航流程图（Mermaid），将新页面标注为"本轮新增"
   - 如果涉及交互变更，更新交互原型
   - **用户确认新页面的导航流与布局后再编码**

5. **进入 Step 5-8 的正常流程**：
   - Step 5: 只拆解本轮迭代涉及的任务
   - Step 6: 正常开发
   - Step 7: 必须包含页面完整性检查（对照本轮定义的范围）
   - **Step 7.3 契约同步与文档对齐检查（v2.1.0 新增）**
     - 增量开发也必须经过同样的门禁检查
   - Step 8: 正常收口
   - **Step 8.2 修复验证（v2.1.0 新增）**
     - 增量开发也必须运行 pytest + npm run build
