# 自动化测试交付流水线 (Automated Test Lifecycle)

这是一个标准的测试交付流水线 (SOP)。当用户提供新的需求或更新的特性时，按照以下步骤严格执行：

1. **获取信息与需求审计 (只做审计)**：
   - 首先收集用户提供的需求描述内容。
   - 调用 `requirement-reviewer` Skill，重点审查逻辑真空以及 P3 / P4 / P5 风险。
   - **【落地建议 4：多存中间文件】**：将生成的审计表格保存为临时文件 `/tmp/requirement_audit.md`。
   - 向用户汇报并展示该文件，询问："您需要补充相关异常情况的处理逻辑，还是直接基于当前设定生成测试策略？"
   - **【验证标准】**：等待用户明确回复"继续"或解决掉高优风险点后再继续。

2. **建立可追溯测试架构 (只做骨架建模)**：
   - 读取刚才的审计结果，调用 `traceability-test-architecture` Skill。
   - 基于当前规格来源建立测试骨架，至少明确：
     - `TC-Px-yyy` 编号规则
     - `traceability.yaml` 的最小字段和维护口径
     - `unit / contract / integration / resilience / golden` 分层方案
     - 分层编排脚本骨架（如 `qa-unit`、`qa-contract`、`qa-all`、`qa-audit`）
     - 前端 E2E 编号要求
     - 统一审计入口与 TDD 门禁要求
   - 将骨架设计保存到临时文件 `/tmp/traceability_architecture.md`。
   - 向用户汇报并展示该文件，询问："是否按这套可追溯测试架构继续展开测试策略与用例设计？"
   - **【验证标准】**：等待用户明确确认后，再进入下一步；若用户要求裁剪层级或调整编号体系，先更新骨架再继续。

3. **输出全景测试策略 (只做架构设计)**：
   - 读取刚才的审计结果，调用 `test-strategy-planner` Skill。
   - 优先将 `/tmp/traceability_architecture.md` 作为上游约束输入，确保策略矩阵与编号体系、测试分层和审计口径一致。
   - 严格按照 P1 到 P5 输出覆盖测试全场景的矩阵策略大纲。
   - **【落地建议 4：多存中间文件】**：将策略大纲保存到临时文件 `/tmp/test_strategy_draft.md`。
   - 将策略大纲展示给用户，并说明："这是为您设计的全局测试策略网，如果方向无误，我将开始为您大规模生成详细用例。"
   - **【验证标准】**：等待用户确认（Approval）矩阵内容后，再授权执行下一步。

4. **批量生成标准测试用例 (只做执行)**：
   - **【落地建议 2：Skill 原子化】**：严格读取 `/tmp/test_strategy_draft.md` 作为唯一输入上下文，调用 `test-case-factory` Skill。
   - 严格参考项目约定的用例模板，生成符合 `TCS` 规范的标准测试用例（包括明确的隔离配置前提和报错验证要求）。
   - 将生成的用例表格直接写入到正式文档 `docs/test_specification.md` 分栏中，并要求用户前往复核。
   - 如当前任务只是补充字段级边界值或行业计划文档，不要偏航调用 `bva-boundary-value-analysis` 或 `test-plan-copilot` 来替代本主链步骤。

4.5 **哑数据 (Mock/Dummy) 契约清场扫描 (防腐验证)**：
   - 在进入下一轮最终复核或发布投产前，必须执行静态代码扫描与核查。
   - 目标是发现并未被消费的空接口、写死的占位假按钮、以及为规避真实契约而保留的骨架代码（如硬编码文件类或未打通后端的死逻辑）。
   - **【落地建议 5：强制 TDD】**：本步骤强推 **Design -> Test -> Code** 编程纪律，确保只有满足真实验收逻辑流转（非造假）的代码才能放行，将未被替换的假壳子物理阻截在本次发布之外。

5. **全链路集成测试 (v2.2.0 新增)**：
   - **【强制】** 每个涉及多模块交互的特性，必须编写全链路集成测试。
   - **后端集成测试**：使用 `httpx.AsyncClient` + in-memory SQLite，串联完整 API 调用链（创建→上传→OCR→审核→查询），mock 外部依赖（OCR 服务），但走真实代码路径。使用真实测试图片（`backend/tests/01.jpg`），禁止使用 `b"dummy-content"` 假数据。
   - **前端 E2E 测试**：使用 Playwright 驱动浏览器，通过真实后端完成端到端验证。测试开头必须包含环境自检（`fetch('/health')`，不通则 `test.skip()`）。
   - **移动端全链路测试**：使用 mock fetch 串联服务层调用，覆盖正常链路 + 异常链路 + 幂等 + 弱网 + 数据污染。

6. **失败路径测试 (v2.2.0 新增)**：
   - **【强制】** 每个关键操作必须覆盖以下异常场景：
     - **弱网**：超时、断线重连、慢响应后重试成功
     - **幂等性**：重复上传/重复提交/重复审核不产生副作用
     - **服务端错误**：500/502/503 后的重试与放弃策略
     - **数据污染防护**：部分失败不产生脏数据（如上传成功但 OCR 失败，Dashboard 应显示空数据）

7. **代码审查 (v2.2.0 新增)**：
   - **【强制】** 在代码变更完成后、commit 之前，必须执行代码审查。
   - **审查范围**：所有新增/修改的文件，重点关注：
     - 逻辑错误（三元表达式两分支相同、数组索引越界、未排序取最新值）
     - 契约一致性（API 路径、响应字段、枚举值）
     - 硬编码常量（localhost、固定端口、绝对路径）
     - 未使用的 import/变量、重复代码
   - **审查方式**：使用 `git diff` 查看所有变更，逐个文件阅读完整内容（不能只看 diff）。
   - **发现问题后**：先写失败的 UT 确认 bug 存在，再修复，再验证通过（TDD 纪律）。

8. **Reviewer Agent 最终复核 (只做收口)**：
   - 在正式交付前，调用 `reviewer-agent` Skill，对当前 workflow 产出的 QA 正式资产进行收口复核。
   - **优先组合**：若环境已安装 `vibe-tools`，使用 `vibe-tools repo` 获取整仓上下文，并明确要求只检查 QA 工作流编排、测试资产完整性、文档同步性与技能治理合规性；若仅复核本轮改动，优先追加 `--with-diff`。
   - 重点检查：
     - `traceability.yaml`、TC 编号、测试分层和统一审计口径是否已经与本轮规格和测试产出对齐。
     - 新增或更新的测试用例、测试报告或缺陷单是否与前述测试策略、模板约束和隔离前提一致。
     - `README.md`、新手指南、用户指南与架构说明是否仍停留在旧流程。
     - 新增 Skill 是否补齐中文说明、版本号与最后更新日期，且没有与现有 QA 资产命名冲突。
     - `.repomix-output.txt`、临时导出文本、调试截图索引等工具运行产物是否被误当成正式资产或纳入提交范围。
   - `test-code-simplifier` 仅在用户明确要求整理测试代码时，才可作为本主链之外的可选收口工具；不得默认插入主流程。
   - 将复核结论输出为"必须修正 / 建议优化 / 可接受风险"三段式摘要；若存在阻断性问题，必须先修正再进入提交阶段。
