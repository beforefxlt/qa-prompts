# Factory Inspector 架构设计思想

本文档总结了 `factory_inspector` 的核心架构设计理念，旨在为后续功能扩展和类似工具的设计提供参考。

## 1. 核心原则：插件化 (Plug-in)
项目采用了**微内核 + 插件**的结构。核心引擎（Engine）仅负责生命周期调度，而不包含任何具体的业务检测逻辑。
- **抽象解耦**：通过 `BasePlugin` 定义标准接口，引擎通过多态方式调用 `run()`。
- **动态扫描**：利用 Python 的动态特性（`importlib`, `inspect`），实现了插件的“即插即用”，无需在核心代码中注册。

## 2. 配置驱动 (Configuration-Driven)
工具的所有行为（测什么、通过标准是什么）都由外部的 `config.yaml` 驱动。
- **阈值管理**：插件不硬编码具体数值，而是从配置片段中读取。
- **逻辑通用化**：同一套代码可以通过不同的配置文件，适配不同型号、不同交付标准的机器。

## 3. 标准化输出 (Standardized Results)
通过 `CheckResult` 数据类强制规范化所有检测项的返回格式。
- **统一展现**：无论是硬件还是网络检测，上报层（Reporter）看到的都是一致的 `status`, `expected`, `actual` 元数据。
- **可观测性**：标准化的输出使得生成 JSON 报告、推送到 MES 系统或进行大数据分析变得异常简单。

## 4. 健壮性与兼容性设计
- **防御性探测**：插件在调用系统命令（如 `ip`, `df`）时，通过 `try...except` 捕获异常，将基础设施错误（如命令缺失）转化为合规性失败结果，而非导致程序崩溃。
- **向下兼容 (Python 3.8)**：考虑到 Ubuntu 20.04 的默认环境，通过 `from __future__ import annotations` 解决了 PEP 585 类型标注在旧版本 Python 中的解析问题。

---

## 5. 适用场景总结
该框架非常适合以下特征的任务：
- **规则多样化**：需要执行大量互相独立的校验规则。
- **环境多变**：需要在不同的操作系统版本或硬件配置下重复运行。
- **高度定制化**：用户需要根据不同项目快速增加特定的检测逻辑。
