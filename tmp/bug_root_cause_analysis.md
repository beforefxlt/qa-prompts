# 缺陷根因隔离分析 (Root Cause Analysis)

依据 `/tmp/bug_investigation_raw.md` 记录的事实进行推演：

### 1. D1-D5 维度诊断
- **D1 (日志与特征)**：日志仅有纯粹的 `Timeout` 而非报错响应（如 Exception/NACK），说明网关发出了完整的 Modbus RTU 报文，但在设定的等待时间内，总线上完全没有收到属于 2 号和 3 号从站的有效回传信号。
- **D3 (环境与物理链路)**：采用 RS485 串行总线（手拉手拓扑），波特率 9600 bps。第 1 个电表正常，说明**网关主栈本身正常，电平转化及前半段物理线路正常**。只有后半段设备失联，典型属于现场环境与网关定序的匹配问题。
- **D4 (边界与时序)**：虽然轮询间隔是宽松的 3 秒，但这通常是“两轮完整采集”的间隔。关键边界在于“**单个报文响应超时（Response Timeout）**”设定。波特率 9600 较低，传输大块报文天然需要较多时间（约 1ms/byte）。如果网关的 Response Timeout 设得太严（如 100ms），或者针对并发或串行队列没有做良好的总线互斥控制（在第一个表还没释放总线控制权时就去寻址第二个表），就会直接引发后续超时。

### 2. 根因推断
- **疑似方向 A（硬件部署问题）**：最典型的 P5 交付问题——从第 2 个表开始，RS485 总线的 A/B 线接反，或者终端表未通过拨码/屏幕正确设置 Modbus Slave ID，导致网关寻了个根本不存在的空地址，必然 `Timeout`。
- **疑似方向 B（时序边界冲突）**：网关的轮询逻辑可能在“多从站串行读取”时未引入合理的延时或总线清空机制（Inter-frame Delay）。某些低端电表处理性能差，在应答后未及时将总线收发器切换回接收状态，引发后续通讯的微秒级冲突。
- **疑似方向 C（分包读取限制过载）**：第 2、3 号表的点表配置如果要求单次读取超过 125 寄存器（或表计不支持的多字读取），表端在接收到不合法请求时往往执行静默处理（不回异常报文），这就表现成了针对特定设备的“纯超时”。

### 3. 系统影响
- **功能性影响**：系统丢失 2/3 的表计遥测和计费数据，电表采集功能处于残缺运行状态。

## 严重程度定级 (Severity Triage)

- **等级**：**P1 - 严重缺陷**
- **命中因子**：**因子 4 - 功能正确性影响**
- **判定依据**：该缺陷不直接引发现场起火或失控等安全风险，且未引发系统宕机。但它导致了核心业务流水（电表遥测或结算数据采集）的直接缺失，干扰了系统的数据准确性。由于现场可能需要人工排查布线或改配置（临时 Workaround 是单独接入），在正式体系中应定级为功能性阻断缺陷 P1。
