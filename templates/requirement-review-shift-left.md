# 需求可测试性与歧义审查（Shift-Left Testing）

**版本**: v1.0
**适用场景**: 需求评审、PRD 审计、用户故事验收
**模板编号**: REQ-001

---

## 模板定义

```markdown
# Role: 资深需求分析专家 / 测试架构师

# Context:
当前处于项目早期评审阶段。我们需要对以下 [需求描述/用户故事] 进行深度审计，
以减少后期因需求模糊导致的返工。

# Raw Requirement:
{{requirement_text}}

# Task:
请从测试视角对上述需求进行"压力测试"，输出评审报告。

# Audit Dimensions:
1. 歧义检测：找出类似"快速"、"稳定"、"大量"、"优化"等缺乏具体指标的词汇
2. 逻辑真空：识别需求中未定义的异常路径（侧重 P0 基建异常、P2 业务异常）
3. 一致性检查：检查该功能与系统现有逻辑是否冲突
4. 可测量性：评估是否具备明确的验收标准（Acceptance Criteria）

# Output Format:
| 需求点 | 潜在风险/歧义 | 建议优化方案 | 风险等级 |
| :--- | :--- | :--- | :--- |
```

## 使用示例

### 示例 1：电商促销需求审查

**输入变量**：
```
{{requirement_text}} =
"用户可以在双十一期间领取优惠券，优惠券可以在下单时抵扣。
系统需要快速响应用户的领券请求，保证用户体验。
优惠券有多种类型，包括满减券、折扣券等。
商家可以设置优惠券的使用门槛。"
```

**预期输出**：

| 需求点 | 潜在风险/歧义 | 建议优化方案 | 风险等级 |
| :--- | :--- | :--- | :--- |
| "快速响应" | 歧义词，无具体指标 | 明确为"95% 的领券请求响应时间<200ms" | 中 |
| "多种类型" | 未列举完整类型 | 枚举所有优惠券类型及优先级规则 | 高 |
| "使用门槛" | 未定义门槛计算方式 | 明确门槛是否包含运费、是否可叠加 | 高 |
| 未定义异常路径 | 库存已领完时如何处理？ | 补充"领完即止"的提示文案和前端状态 | 中 |
| 未定义异常路径 | 优惠券过期后的处理逻辑 | 补充过期自动作废、不可恢复的说明 | 中 |
| 一致性风险 | 与现有会员折扣是否叠加？ | 明确优惠券与其他优惠的优先级 | 高 |
| 可测量性不足 | "多种类型"缺乏上限 | 明确单个商家最多创建几种优惠券 | 低 |

---

### 示例 2：用户积分系统需求审查

**输入变量**：
```
{{requirement_text}} =
"用户每天签到可以获得积分，积分可以兑换礼品。
连续签到会有额外奖励。
积分有有效期，过期作废。
系统要防止用户刷积分。"
```

**预期输出**：

| 需求点 | 潜在风险/歧义 | 建议优化方案 | 风险等级 |
| :--- | :--- | :--- | :--- |
| "每天签到" | 自然日还是 24 小时？时区如何定义？ | 明确为"自然日 00:00-23:59(UTC+8)" | 中 |
| "连续签到" | 中断后如何计算？ | 明确"断签一天后，连续计数清零" | 高 |
| "额外奖励" | 未定义奖励规则和上限 | 补充连续 7 天、30 天的具体奖励数值 | 高 |
| "有效期" | 未说明起算时间和周期 | 明确"积分自获得日起 12 个月有效" | 高 |
| "防止刷积分" | 缺乏具体风控措施 | 补充设备指纹、IP 限制、行为分析规则 | 高 |
| 逻辑真空 | 签到服务异常时的补偿机制 | 补充异常补签流程和限制条件 | 中 |
| 一致性风险 | 积分扣减为负时的处理 | 明确"积分不足时兑换失败"的提示 | 中 |

---

### 示例 3：数据导出功能需求审查

**输入变量**：
```
{{requirement_text}} =
"管理员可以导出用户数据报表。
系统支持大量数据的导出。
导出文件应该包含所有相关信息。
为了安全，需要权限控制。"
```

**预期输出**：

| 需求点 | 潜在风险/歧义 | 建议优化方案 | 风险等级 |
| :--- | :--- | :--- | :--- |
| "大量数据" | 无具体数量级定义 | 明确"支持最多 10 万条记录导出" | 高 |
| "所有相关信息" | 字段定义不明确 | 附完整字段清单及排序规则 | 高 |
| "权限控制" | 未说明权限粒度 | 明确角色权限矩阵（谁能导出什么数据） | 高 |
| 逻辑真空 | 导出超时如何处理？ | 补充"异步导出 + 邮件通知"机制 | 中 |
| 逻辑真空 | 导出过程中数据变更的处理 | 明确"快照时间戳"和一致性保证 | 中 |
| 可测量性不足 | 导出性能无指标 | 补充"10 万条数据导出<5 分钟" | 中 |
| 安全风险 | 敏感字段是否需要脱敏 | 补充手机号、身份证的脱敏规则 | 高 |

---

## 风险等级定义

| 等级 | 说明 | 处理建议 |
| :--- | :--- | :--- |
| **高** | 可能导致严重功能缺陷或返工 | 必须在开发前澄清 |
| **中** | 可能影响用户体验或增加测试成本 | 建议在迭代内澄清 |
| **低** | 边缘情况，影响有限 | 可记录待办，后续优化 |

---

## 审查检查清单

使用此模板时，请确保覆盖以下维度：

- [ ] **歧义词检测**：快速、稳定、大量、实时、友好、智能
- [ ] **数值指标**：是否有明确的数字、时间、百分比
- [ ] **异常路径**：超时、失败、空值、重复、并发
- [ ] **边界条件**：最大值、最小值、零值、负值
- [ ] **状态一致性**：与现有功能是否有冲突
- [ ] **权限定义**：谁能做什么，不能做什么
- [ ] **数据规则**：增删改查的完整定义
- [ ] **验收标准**：是否可测试、可度量
