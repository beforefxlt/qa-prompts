## 一、测试架构与分层策略

### 1.1 测试金字塔模型

```
                    ┌─────────────────┐
                    │    E2E Tests    │  ← L4: 用户场景 + 路径完整性
                    │   (Playwright)  │     执行：慢 | 成本：高
                    └────────┬────────┘
                             │
                ┌────────────┴────────────┐
                │    Contract Tests       │  ← L3: API 契约校验 (Schema)
                │    Integration Tests    │  ← L2: 接口回归相关
                │      (Python API)       │     执行：中 | 成本：中
                └────────────┬────────────┘
                             │
           ┌─────────────────┴─────────────────┐
           │          Unit Tests               │  ← L1: 单元测试
           │ (Go Testing + Jest / JSDOM)       │     执行：快 | 成本：低
           └───────────────────────────────────┘
```

### 1.2 各层级详解

| 层级 | 测试内容 | 工具/脚本 | 执行入口 |
| :--- | :--- | :--- | :--- |
| **L1: 单元测试** | 路由、中间件、LaTeX 清洗、重试/降级逻辑 | Go `testing` | `go test ./...` |
| **L1.6: 前端 UT** | DOM 模板、Timeout 降级、500 逻辑覆盖 | Jest + JSDOM | `npm test` |
| **L2: 接口回归** | 真实图片 OCR + 批改全链路验证 | pytest | `tests/functional/test_regression.py` |
| **L3: 契约校验** | JSON Schema 结构与字段合法性校验 | pytest | `tests/functional/test_contract.py` |
| **L4: 格式审计** | AI 输出稳定性（Emoji、LaTeX 语法） | Python | `analyze_prob.py` |
| **L5: 场景 E2E** | 模拟"拍照 - 裁剪 - 等待 - 结果"全流程 | Playwright (Python) | `tests/e2e/test_real_scenario.py` |

### 1.3 核心原则

- **快速反馈**：L1 单元测试应在秒级完成
- **三位一体覆盖**：每个 API 调用必须覆盖 Happy Path、Logic Error 和 Transport Error
- **协议仿真**：使用 `httptest` 进行 URL 劫持，不产生真实 HTTP 出站请求
- **防御性要求**：Wait-Check-Access 模式的实操检验

## 四、AI 驱动开发的测试约束

### 4.1 单元测试生成约束

**输入数据保真度**：
- 不要简化 `Arrange` 输入，必须使用 messy、raw strings
- 禁止使用人类精简过的理想数据（如 `{"key": "value"}`）
- 必须测试包含 Markdown wrapper、LaTeX blocks 的真实 AI 输出

**断言强度分级**：

| 强度 | 定义 | 示例 | 可接受用途 |
|------|------|------|-----------|
| **L0** | 存在性 | `count > 0` | 仅冒烟测试 |
| **L1** | 非空 | `len(title) > 0` | 不可用于业务逻辑 |
| **L2** | 类型 | `assert.IsType()` | 仅作中间保护 |
| **L3** | 值域 | `count >= 2` | 业务逻辑最低标准 |
| **L4** | 精确匹配 | `assert.Equal(3, count)` | 推荐 |

**断言红线（直接打回）**：
- `json.Unmarshal(...)` error 被 `_` 丢弃
- 类型断言 `.(Type)` 无 `ok` 检查
- E2E 阈值 `>= 1` 用于多元素场景
- 用 `log` 代替 `assert`

### 4.2 穿透式契约扫描

```
┌─────────────────────────────────────────────────────────┐
│  隔离测试造假：每个函数分别有自己的 UT，但没人测试     │
│  把文本先传给 A，再传给 B 会有什么结果                   │
│  (L1 层极度冗余，L3 模块集成层为 0)                     │
└─────────────────────────────────────────────────────────┘
```

**解决方案**：
1. Input 必须直接复制自物理真实大模型输出的脏数据长串
2. 强制调用总入口 (Handler/Service) 的契约测试
3. 模拟带有真实物理特征的脏字符串流动

### 4.3 隔离越权能力

- 设计层面将 `纯文本 OCR 清洗` 和 `JSON 强制提取` 切割为两个物理隔绝的通路
- 集成测试证明：Route A 的 clean inputs 不被 Route B 的过度清洗规则破坏

## 五、测试工程化治理

### 5.1 单一事实来源 (SSOT)

**问题**：配置分散导致漂移（本地测试正常，CI 静默失败）

**解决方案**：

```python
# run_tests.py 中声明 Provider 清单（唯一事实来源）
PROVIDERS_IN_USE = [
    {"key": "SILICONFLOW_API_KEY", "role": "OCR + 判卷", "tier": "primary"},
    {"key": "NVIDIA_API_KEY",      "role": "Fallback 判卷", "tier": "fallback"},
    {"key": "ZHIPU_API_KEY",       "role": "Fallback OCR", "tier": "fallback"},
]
```

```yaml
# CI yaml 只做 Secret 传递，不写业务判断
env:
  SILICONFLOW_API_KEY: ${{ secrets.SILICONFLOW_API_KEY }}
  NVIDIA_API_KEY:      ${{ secrets.NVIDIA_API_KEY }}
  ZHIPU_API_KEY:       ${{ secrets.ZHIPU_API_KEY }}
```

### 5.2 Pre-commit 守护

```python
# check_ci_provider_sync()
# 1. 读取 PROVIDERS_IN_USE
# 2. 解析 ci-cd.yml 中的 env 块
# 3. 比对两者 key 列表
# 4. 发现漂移则阻断提交并提示修复
```

**效果**：
- 新增 Provider：改清单 → Pre-commit 提示补 CI yaml → 补 Secret
- 删除 Provider：移除清单 → CI yaml 遗留无害
- 永不漂移：提交钩子本地拦截不一致

### 5.3 工具层封装 (Test Utilities)

统一 `APIClient` 职责：
- 鉴权注入
- 自动重试
- 超时控制
- NO_PROXY 设置

### 5.4 报告聚合

跨语言测试产出统一 JSON 格式：
- `summary.json` - 顶层汇总
- `index.html` - 可视化大屏
- `allure-report/` - 交互式报告

### 5.5 日志可见性设计

**双重性格执行**：

| 模式 | 行为 | 目的 |
|------|------|------|
| 执行时静默 | 捕获 stdout/stderr，只显示进度 | 避免日志淹没 |
| 失败时爆破 | 保存输出到 report.output | 保留现场 |
| 底部汇总 | 打印 `Error Summary: Last 30 lines` | 快速定位根因 |

**核心教训**：测试失败的第一秒，开发者就应该看到 Root Cause。

## 七、断言质量保障

### 7.1 AI 断言 5 大通病

| 通病 | 表现 | 修复方案 |
|------|------|----------|
| "不崩即过" | 只检查不报错 | 绑定业务预期 |
| 阈值偷懒 | `> 0` 代替 `>= 2` | 精确阈值 |
| error 丢弃 | `_ = json.Unmarshal()` | 强制检查 error |
| 类型裸奔 | `.(Type)` 无 `ok` | 安全断言 |
| 存在性代替正确性 | 检查元素存在 | 检查内容正确 |

### 7.2 变异测试思维

审查每条断言时问：
> "如果我故意把被测代码改错，这条断言能抓住吗？"

如果答案是"不能"，这条断言就是无效的。

### 7.3 人工审查红线（直接打回）

- `json.Unmarshal()` error 被 `_` 丢弃
- 类型断言无 `ok` 检查
- E2E 阈值 `>= 1` 用于多元素场景
- 用 `log` 代替 `assert`
- 只检查 HTTP 状态码不检查 Body 内容

### 7.4 为什么 Go 后端测试质量高于 E2E？

| 因素 | Go 后端 UT | E2E (JS/Python) |
|------|-----------|-----------------|
| **语言类型系统** | 编译期强类型 | 弱类型，运行时暴露 |
| **AI 训练语料** | Go testing 范例规范 | Playwright 场景多样 |
| **断言粒度** | 精确到字段值 | 倾向于存在性检查 |
| **反馈速度** | 编译不过就报错 | 启动浏览器才能发现 |

## 八、实战反思录

### 8.1 500 Panic 教训

**事件**：在检查 error 前访问了 nil 响应，导致服务崩溃

**防御模型**：Wait-Check-Access-Guard
1. Wait - 发起请求
2. Check - 第一时间检查 err
3. Guard - 确认指针非空 (`if resp != nil`)
4. Access - 读取数据

### 8.2 前端 UT 盲区

**症状**：前端代码"零单元测试，全靠 E2E 兜底"

**根因**：
- Vanilla JS 强耦合 DOM，AI 习惯性将业务逻辑和 DOM 操作耦合
- 搭建 Jest + JSDOM 环境繁琐，AI 倾向写 Playwright 脚本

**对策**：
- PRD 阶段强制要求纯函数隔离
- 引入 Jest + JSDOM 秒级反馈（< 30ms）
- 可测试性约束写入文档

### 8.3 E2E 断言放水

**症状**：测试全绿但功能异常（多题手风琴样式没出现）

**根因**：断言阈值与业务规格脱节

| 漏洞 | 说明 |
|------|------|
| `#result-content` 存在性判断无效 | 该 div 写死在 HTML 中，永远存在 |
| API 拦截数量未参与判定 | 只打 log，不影响 pass/fail |
| DOM 卡片阈值过低 | `renderedCount >= 1` 即通过，单题也能过关 |

**修复**：
- API 拦截数量 `capturedResultsCount >= 2` 且参与判定
- DOM 层 `renderedCount >= 2`
- 摘要正则提取"共发现 N 道题目"中的 N，要求 N >= 2

### 8.4 管道集成的漏网之鱼

**症状**：单元测试全绿，集成后 Bug 频发（cleanJSON 越权清洗）

**根因**：测试避开真实清洗管道

**对策**：
- Input 必须使用真实 AI 输出的脏数据长串
- 强制穿透式集成点测试
- 消除越权函数职责

### 8.5 CD 部署"假成功"

**Bug 链条**：
1. systemd 路径不一致 → 重启的是旧二进制
2. pkill 进程名不匹配 → 旧进程占端口
3. ldflags 未注入 Key → 新二进制因无 Key 而 500

**教训**：部署成功 ≠ 服务生效。必须有端到端健康检查。

## 九、CI/CD 集成指南

### 9.1 流水线阶段

| 阶段 | 执行内容 | 失败处理 |
|------|----------|----------|
| Checkout | 拉取代码 | 重试 |
| Setup | 安装 Go/Node/Python/依赖 | 失败终止 |
| Unit Tests | `go test ./...` + Jest | 失败终止 |
| Integration Tests | 回归 + 契约 + E2E | 失败终止 |
| Build | 交叉编译 | 测试通过后执行 |
| Artifact | 上传 Release 包 | - |

### 9.2 覆盖率门禁配置

```yaml
- name: Python Coverage Gate
  run: |
    pytest tests/unit/ \
      --cov=scripts \
      --cov=tests \
      --cov-report=xml:coverage_python.xml \
      --cov-fail-under=70

- name: Go Coverage Gate
  run: |
    go test -count=1 -coverprofile=coverage_go.out -covermode=atomic ./...
    COVERAGE=$(go tool cover -func=coverage_go.out | grep total | awk '{print $3}' | tr -d '%')
    if (( $(echo "$COVERAGE < 70" | bc -l) )); then
        echo "Go 覆盖率 $COVERAGE% 低于 70% 门禁"
        exit 1
    fi
```

### 9.3 部署脚本关键实现

**进程清理组合拳**：
```bash
# 先温和停止，再强力清除，最后清理端口
systemctl stop study-inspire || true
killall -9 study-inspire-linux-amd64 || true
fuser -k 8080/tcp || true
```

**密钥编译时注入**：
```python
# 从 .env 读取 Key 注入到二进制 (双保险)
ldflags_pairs = []
for line in open(".env"):
    if "=" in line and not line.startswith("#"):
        k, v = line.split("=", 1)
        if k in key_mapping:
            ldflags_pairs.append(f"-X config.{key_mapping[k]}={v}")
ldflags = " ".join(ldflags_pairs)
build_cmd = ["go", "build", "-o", binary_path, "-ldflags", ldflags, "."]
```

### 9.4 健康检查

```python
# 检查 1: 后端 /ping
resp = urllib.request.urlopen(f"http://{SERVER_IP}:8080/ping", timeout=10)
if b"pong" in resp.read():
    print("[PASS] /ping -> pong")

# 检查 2: 前端关键 DOM
html = urllib.request.urlopen(f"http://{SERVER_IP}:8080/").read().decode()
required = ["result-content", "app.js"]
missing = [el for el in required if el not in html]
if not missing:
    print("[PASS] Frontend OK")
```
