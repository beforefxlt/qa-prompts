# 新系统测试设计经验指南

> 从 StudyInspire 项目实战中提炼的通用测试架构设计指南

---

## 一、核心理念

### 1.1 测试金字塔经济学

```
                    ┌─────────────────┐
                    │    E2E Tests    │  置信度高，成本高，执行慢
                    └────────┬────────┘
                ┌────────────┴────────────┐
                │   Contract + Integration│  置信度中，成本中，执行中
                └────────────┬────────────┘
           ┌─────────────────┴─────────────────┐
           │          Unit Tests               │  稳定性高，成本低，执行快
           └───────────────────────────────────┘
```

**核心原则**：能在底层 UT 测清的逻辑，绝不推到高层集成测试。

### 1.2 AI 时代的测试挑战

| 挑战 | 解决方案 |
|------|----------|
| 输出不确定性 | 概率分析 + Sanitizer 后处理 |
| 测试代码漂移 | SSOT + Pre-commit 自动校验 |
| 断言放水 | 变异测试思维 + 强度分级审查 |
| 环境不一致 | 编译时注入 + .env 统一配置 |

## 二、测试分层设计 (L1-L6)

### L1: 单元测试 (Unit Tests)

**目标**：毫秒级反馈，覆盖纯逻辑

**必须覆盖的场景**：
1. **Happy Path** - 正常返回 200
2. **Logic Error** - 4xx/5xx 错误处理
3. **Transport Error** - nil response，杜绝空指针 Panic

**协议仿真原则**：使用 `httptest` 劫持传输层，而非简单 Mock 接口。

### L2: 集成测试 (Integration Tests)

**目标**：验证模块间接口握手

**关键实践**：
- 真实 Router + Middleware + Handler 整合
- 内存事务回滚 (`defer harness.Teardown()`)
- 禁止状态污染

### L3: 契约测试 (Contract Testing)

**目标**：防止前后端数据结构漂移

**SSOT 原则**：
- 在 `api_schema.py` 定义 JSON Schema
- 与后端 Go Structs 严格对应
- 增加业务语义校验（如禁止特殊 LaTeX 符号）

### L4: 格式审计 (Stability Checks)

**目标**：验证 AI 输出稳定性

**检查项**：
- 禁止字符检测（Emoji、非法 LaTeX 命令）
- 概率性测试（连续请求 N 次统计失败率）

### L4: E2E 场景测试（推荐合并 L5/L6）

**目标**：模拟用户真实操作流程，验证端到端业务链路

**断言强度要求**（不得低于 L3）：

| 强度 | 定义 | 示例 |
|------|------|------|
| L0 存在性 | 元素存在 | `count > 0` |
| L3 值域 | 值在业务范围内 | `count >= 2` |
| L4 精确匹配 | 完全等于预期 | `assert.Equal(3, count)` |

**执行模式**：
```python
# 轻量模式：本地开发，1 个场景，30s 完成
pytest tests/e2e/test_real_scenario.py -k "single"

# 深度模式：CI 运行，3 个场景，120s 完成
pytest tests/e2e/ -m "full"
```

## 三、测试分层合并建议（重要）

### 合并 L5 + L6 为统一 L4 的原因

本项目原设计将 E2E 分为 L5（Python Playwright）和 L6（Node.js Playwright），但经过实战验证，**建议合并为统一层**：

| 对比维度 | L5 (`test_real_scenario.py`) | L6 (`e2e_web_test.js`) | 合并理由 |
|----------|------------------------------|------------------------|----------|
| **工具栈** | Playwright (Python) | Playwright (Node.js) | 同一框架，无需双栈维护 |
| **执行目的** | 单场景冒烟 | 多场景 + 渲染质量 | 可通过参数/标记区分 |
| **断言深度** | 基础存在性 | API 拦截 + 质量校验 | 可在同一脚本分层实现 |
| **维护成本** | 独立维护 | 独立维护 | 重复代码 > 30% |

**合并判断标准**：

| 标准 | 本项目情况 | 结论 |
|------|------------|------|
| 执行目的不同 | 都是验证 E2E 链路 | ❌ 不满足 |
| 执行频率不同 | 都每次 CI 运行 | ❌ 不满足 |
| 维护成本可控 | 两套脚本独立维护 | ❌ 不满足 |

**推荐架构**：

```
L1: Unit Tests          ← 毫秒级，纯逻辑
L2: Integration Tests   ← 验证模块握手
L3: Contract Tests      ← Schema 校验
L4: E2E Tests           ← 统一 Playwright，分轻量/深度模式
```

## 三、测试工程化治理

### 3.1 单一事实来源 (SSOT)

**问题**：配置分散导致漂移（本地测试正常，CI 静默失败）

**解决方案**：

```python
# run_tests.py 中声明 Provider 清单（唯一事实来源）
PROVIDERS_IN_USE = [
    {"key": "SILICONFLOW_API_KEY", "role": "OCR + 判卷", "tier": "primary"},
    {"key": "NVIDIA_API_KEY",      "role": "Fallback 判卷", "tier": "fallback"},
]
```

```yaml
# CI yaml 只做 Secret 传递，不写业务判断
env:
  SILICONFLOW_API_KEY: ${{ secrets.SILICONFLOW_API_KEY }}
  NVIDIA_API_KEY:      ${{ secrets.NVIDIA_API_KEY }}
```

### 3.2 工具层封装 (Test Utilities)

统一 `APIClient` 职责：
- 鉴权注入
- 自动重试
- 超时控制
- NO_PROXY 设置

### 3.3 报告聚合

跨语言测试产出统一 JSON 格式，生成：
- `summary.json` - 顶层汇总
- `index.html` - 可视化大屏
- `allure-report/` - 交互式报告

## 四、断言质量保障

### 4.1 AI 断言 5 大通病

| 通病 | 表现 | 修复方案 |
|------|------|----------|
| "不崩即过" | 只检查不报错 | 绑定业务预期 |
| 阈值偷懒 | `> 0` 代替 `>= 2` | 精确阈值 |
| error 丢弃 | `_ = json.Unmarshal()` | 强制检查 error |
| 类型裸奔 | `.(Type)` 无 `ok` | 安全断言 |
| 存在性代替正确性 | 检查元素存在 | 检查内容正确 |

### 4.2 变异测试思维

审查每条断言时问：
> "如果我故意把被测代码改错，这条断言能抓住吗？"

### 4.3 人工审查红线（直接打回）

- `json.Unmarshal()` error 被丢弃
- 类型断言无 `ok` 检查
- E2E 阈值 `>= 1` 用于多元素场景
- 用 `log` 代替 `assert`
- 只检查 HTTP 状态码不检查 Body 内容

## 五、实战反思录

### 5.1 500 Panic 教训

**原因**：检查 error 前访问了 nil 响应

**防御模型**：Wait-Check-Access-Guard
1. Wait - 发起请求
2. Check - 第一时间检查 err
3. Guard - 确认指针非空
4. Access - 读取数据

### 5.2 前端 UT 盲区

**教训**：Vanilla JS 强耦合 DOM，无法隔离测试

**对策**：
- PRD 阶段强制要求纯函数隔离
- 引入 Jest + JSDOM 秒级反馈
- 可测试性约束写入文档

### 5.3 E2E 断言放水

**症状**：测试全绿但功能异常

**根因**：断言阈值与业务规格脱节

**修复**：
- API 拦截数量参与 pass/fail 判定
- DOM 卡片数量强校验（`>= 2`）
- 摘要文本正则提取验证

### 5.4 管道集成的漏网之鱼

**症状**：单元测试全绿，集成后 Bug 频发

**根因**：测试避开真实清洗管道

**对策**：
- Input 必须使用真实 AI 输出的脏数据
- 强制穿透式集成点测试 (Penetration Test)
- 消除越权函数职责

## 六、投入产出比 (ROI) 分析

### 6.1 高 ROI 领域（必须推广）

| 领域 | 理由 |
|------|------|
| 跨 3 模块引用的基础构件 | 网络客户端、日志库、配置加载器 |
| 非结构化数据处理 | 正则提取、JSON 容错、符号转换 |
| 纯函数工具类 | LaTeX 清洗、文本处理 |

### 6.2 "三不测"原则

1. **琐碎逻辑不测** - Setter/Getter、简单常量
2. **第三方库底层不测** - 框架原生行为
3. **UI 瞬时中间态不测** - 复杂算法除外

### 6.3 AI 原生开发的特殊要求

凡涉及 **非结构化数据处理** 的函数，必须实施 **像素级 UT 覆盖**。AI 输出具有概率性，因此解析端的代码必须具备 100% 的确定性。

## 八、代码覆盖率规范 (新增)

### 8.1 覆盖率目标

| 语言 | 短期目标 | 长期目标 | 当前状态 (2026-03-06) |
|------|---------|---------|---------------------|
| Python | 75% | 85% | 66% ⚠️ |
| Go | 75% | 85% | 71.1% ✅ |
| **综合** | **75%** | **85%** | **~68.5%** ⚠️ |

### 8.2 覆盖率门禁

| 检查时机 | Python | Go |
|---------|--------|-----|
| Pre-push | >= 70% | >= 70% |
| PR Check | >= 70% | >= 70% |
| Release | >= 80% | >= 80% |

### 8.3 本地运行覆盖率测试

```bash
# 一键运行 (推荐)
./scripts/coverage_all.sh

# 仅 Python
pytest tests/unit/ --cov=scripts --cov=tests --cov-report=html

# 仅 Go
go test -count=1 -coverprofile=coverage.out -covermode=atomic ./...
go tool cover -html=coverage.out -o coverage.html
```

### 8.4 覆盖率盲区 (可跳过)

以下代码不应强求覆盖率：
1. 配置代码 (`.env` 加载、常量定义)
2. 纯转发/委托方法 (简单的 getter/setter)
3. 第三方适配层 (仅调用外部库的代码)
4. 调试/日志代码 (仅在异常时执行)
5. main 入口函数
6. 自动生成的代码 (protoc 等)

### 8.5 改进优先级

| 优先级 | 模块 | 当前覆盖率 | 目标 | 预计改进 |
|--------|------|-----------|------|---------|
| P0 | `test_utils.py` (Assertions) | 55% | 80% | +25% |
| P0 | `internal/service` | 60% | 80% | +20% |
| P1 | `internal/api` | 61.3% | 80% | +18.7% |
| P2 | `internal/config` | 79.5% | 90% | +10.5% |

## 结语

**测试架构师的价值**：不在于写多少测试用例，而在于设计能够勒住 AI 缰绳的测试防御体系。

**核心公式**：
```
高质量交付 = SSOT(单一事实来源) + 分层防御 + 自动化守护 + 可见性设计
```
