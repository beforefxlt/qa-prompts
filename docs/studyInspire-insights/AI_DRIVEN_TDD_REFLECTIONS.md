# AI 驱动开发下的架构反思：前端 TDD 盲区与 PRD 契约约束

> **背景**: 在 StudyInspire 项目中，尽管早期下达了“基于 TDD 开发，先写测试”的指令，但在很长一段时间内，前端代码（Vanilla JS）处于“零单元测试，全靠 E2E 兜底”的裸奔状态。本文档旨在复盘此现象的根本原因，并提炼出在 AI 驱动开发模式下的工程约束规范。

---

## 1. 核心现象：前端“零单元测试”的长周期盲区

在 AI 驱动开发的模式下，当向 AI 下达宽泛的“TDD”指令时，AI 往往会产生**语义降级**（试图用更为简单的端到端测试来偷换单元测试的概念），从而引发系统性的盲区：

### 1.1 结构性原因：Vanilla JS（原生 JS）的“高耦合陷阱”
- **AI 的本能**：在使用纯 HTML + 原生 JS（如 `app.js`）架构时，AI 习惯于将“业务逻辑（如计算、解析 JSON）”和“DOM 操作（如 `document.getElementById`、`innerHTML` 赋值）”**紧密耦合**在同一个长函数里。
- **UT 的高阻力**：要对这种高耦合代码写单元测试，必须在 Node.js 环境中模拟出一个浏览器（引入 JSOM 和相应的模块化改造）。对 AI 来说，搭建配置繁琐的 Jest + JSDOM 环境，远不如写一段 Python/Playwright 脚本去直接点击浏览器来得“简单粗暴”。

### 1.2 生态不对等：工具链“摩擦力”差异
- **后端的顺滑 TDD**：Go 语言自带 `go test`，零配置。AI 写完后端逻辑即可顺手生成 `_test.go` 并执行。
- **前端的步履维艰**：前端需要 `npm init`、`npm install jest jsdom`、配置 `jest.config.js` 等。在 AI 的多轮对话中，这种基建任务极易丢失或被 AI 刻意回避。

### 1.3 “倒金字塔”幻觉：E2E 掩盖了 UT 的缺位
- **E2E 越俎代庖**：系统被高高在上的 E2E 测试（L4/L5）覆盖。当 E2E 绿灯亮起时，开发者容易产生“代码质量健康”的逻辑错觉。
- **沉默的螺旋**：底层的逻辑错误（如嵌套对象解构错误导致 JS 崩溃白屏）被掩盖，直到最终我们在跑漫长的 E2E 流程或人工测试时才被发现。这种用重器打蚊子的方式，导致了 CI 管线臃肿缓慢，排查成本极高。

---

## 2. 架构师的破局之道：应对“环境知识不对称陷阱”

在全栈或 AI 原生开发中，开发者个人可能存在技术盲区（例如不熟悉前端生态），这导致 AI 在执行宽泛口令时无所适从或选择最偷懒的实现方式。

以下是针对未来新项目或新模块的第一性原理防范策略：

### 2.1 认知重构：PRD 不再仅仅是业务说明，它是“工程约束契约”

传统开发中，技术选型和测试策略由研发内部确认。但在 AI 驱动流程中，**AI 既是产品经理，也是基层码农**。

- **改进必须前移**：在 PRD 的初始结构中，必须雷打不动地包含一个独立的章节：`「技术栈与测试基建约束 (Tech Stack & Testing Infrastructure)」`。

### 2.2 应对盲区：利用“反向索取”让 AI 探路

在对某项技术栈不熟悉时，千万**不要在指令中强行下发具体的框架结论**。相反，应该让 AI 先作为测试架构师进行推演：

> *"我们即将在前端使用 Vanilla JS 进行开发。在此之前，你必须知道我极度看重单元测试（L1 层），且不希望仅仅依赖 Playwright 等 E2E 测试。*
>
> *由于我不熟悉前端测试生态，请你向我输出一份《前端轻量化 UT 实施方案》。方案必须回答：*
> *1. 选用什么框架（如 Jest + JSDOM）？*
> *2. 如何解决原生 JS 中 DOM 操作与业务逻辑解耦的问题？*
> *3. 请给我一个基础的测试用例 Demo。"*

**战略意义**：这迫使 AI 提前承认并接受了“代码必须解耦，并且必须搭好测试脚手架”的工程界桩，斩断了它后续写出强耦合低劣代码的退路。

### 2.3 在 PRD 中确立“面向测试的设计 (Testable Architecture)”

TDD 的核心是**“迫使你写出可测试的代码”**。在未来的大版本 PRD 中，针对全栈盲区需要强行写入以下约束：

1. **纯函数隔离**：前端任何涉及复杂计算、正则提取、文本容错清洗的逻辑，必须剥离为“不依赖 `document` 对象的纯函数 (Pure Functions)”，并单独导出以供测试。
2. **防漂移与秒级测试**：任何前端组件的开发，必须附带一个能脱离浏览器（基于 Node.js/JSDOM）在 **30 毫秒内完成执行**的单元测试。严禁以重型 UI E2E 测试来冲抵单元测试指标。
3. **基建先行**：在完成任何核心业务代码或 DOM 渲染前，必须首先确保测试脚手架环境可成功运转，并以此进行冒烟驱动。

---

## 3. 终极结论

就像制造业中的公差管理一样，全流程 AI 驱动开发的要义是：**先用审查机制把“尺子（测试与规范约束）”造准了，AI 再据此生产出每一个合规的零件（业务代码）。**

## 4. 架构师的新角色：项目初期的基建设计与技术栈选型

> **背景痛点**：在 StudyInspire 项目演进中，我们经历了测试代码散落零星（游击队模式）、本地与 CI 环境变量脱节导致静默失败、以及日志黑洞等问题。对于系统测试背景的架构师而言，如何在开发阶段通过制度和基石设计，利用 AI 能力抵御后期的“环境熵增”，是 AI 时代的重要课题。

### 4.1 测试基建左移：用“物理约束”对抗“AI 的面条式直觉”

如果不做干预，AI 总是倾向于写最快的面条代码（Spaghetti Code）。项目初期的第一步，必须是用物理目录和规则铁壁，把 AI 锁在框架里。

1. **物理目录即架构约束 (Conway's Law for AI)**
   - **分层隔离**：在第一行代码前定死目录标准（如 `tests/unit/` 仅限纯逻辑 Mock 测试，`tests/integration/` 允许网络调用，`scripts/test_infra/` 专属存放 `API_Client` 和公共断言库）。
   - **禁止散兵游勇**：严禁 AI 在单个测试用例中直接使用原生的 `fetch` 或 `requests.get`。当 AI 知道存在 `test_infra` 基建层时，它会自动复用封装好的重试和安全逻辑。

2. **防治“环境熵增”：确立 SSOT（单一事实来源）引擎**
   - **从“人治”到“法治”**：项目后期的配置漂移（如增加了一个备用大模型，但在 GitHub Actions yaml 中忘记配置 Secret）会导致灾难性的“静默降级”。
   - **Pre-commit 强拦截**：我们通过引入 Git Pre-commit 钩子，每次提交流水线前，机器会自动扫描测试脚本中的 Provider 列表与 CI 配置文件是否 1:1 同步。一旦漂移，直接阻断 Commit。用机器的绝对刻板，代替人类记忆的不靠谱。

3. **错误可见性设计：告别查日志玄学**
   - **测试漏斗模式**：测试脚本在正常执行时应保持极度静默（仅进度条），而一旦失败，必须自动拦截底层报错并于执行末尾输出高亮的 `Error Summary: Last 30 Lines`。
   - **AI 的最强弹药**：有了这 30 行精华总结，当 CI 挂掉时，你只需要将 Summary 扔给 AI，它瞬间就能定位是 401 还是 JSON 解析失败，极大地降低了 Token 消耗和排查心智负担。

### 4.2 技术栈选型理念：拥抱主流语料，反向利用 AI 强推最佳实践

系统测试背景的架构师在涉猎开发阶段时，最大的瓶颈往往是对前端或特定框架的代码生态不熟悉。

- **传统思维陷阱**：倾向于使用自己“看得懂”的古老或原生技术栈（例如我们初期使用的 Vanilla JS），认为这样能苟住掌控感。
- **AI 时代的结论**：**必须果断选择业界推荐的主流/现代技术栈（如 React、Vue 或 Go 等结构化框架）。** 传统开发不敢用新框架是怕学习成本；而引入 AI 后，这个学习成本被极大抹平。

#### 为什么要选择主流？—— 因为 AI 吃的是“语料红利”
AI 的能力上限，取决于它在预训练阶段吃过了多少高质量 Github 开源库和 StackOverflow 的最佳实践。
- 当你让 AI 写 React + Jest 时，它脑子里有上百万篇高质量的标准答案，它不仅能帮你搭建优雅的组件，还能极其丝滑地帮你配好 ESLint 和 UT 脚手架。
- 反之，当你让它用原生 JS 去硬干复杂的 DOM 渲染并要求写 UT 时（非主流语境），它会陷入逻辑灾难，不仅代码强耦合，甚至连带起一个 JSDOM 虚拟环境都会错误百出。

#### 如何确保架构师不失控？—— 祭出“反向索取”验证机制
选择了自己不熟悉的主流技术栈后，防失控的最后一道防线是：**在 PRD 阶段向 AI 提出“灵魂三问”**。

向 AI 质询：
1. **“请为我生成这个技术栈最基础的 Hello World 工程，包含完整的自动化构建和包管理配置。”**
2. **“请提供一段核心逻辑如何与 UI 渲染强制解耦的 Demo 代码，向我证明它的可维护性。”**
3. **“请提供一个能在 30 毫秒内跑通的单元测试示例命令，向我证明你的设计是 Testable 的。”**

**总结**：在 AI 开发时代，系统级测试架构师的价值不在于自己手写多少行 React，而在于**“画出最严苛的模具”**。我们利用 AI 对主流框架的先验知识，先逼迫它造出一把符合高标准的“尺子”（自动化测试环境和代码结构规范），随后再放手让它去流水线上生产“零件”（业务特性）。

## 5. 警惕“单元隔离”带来的漏网之鱼：管道集成的契约验证

> **背景痛点**：在 2026-02-26 的一次重构中，爆出了一个深水炸弹 Bug（第32号缺陷）：后端 `cleanJSON` 函数被随意混用于清洗“OCR 纯文本”和“多题批改 JSON 数组”，结果破坏性地把原本正常的文本腰斩、把正确的 JSON 中括号剥去。**令人后怕的是，这个 Bug 在单元测试中竟然长期保持“全绿通过”！**

### 5.1 原因分析：测试代码对生产管道的“完美避让”
回溯代码发现，原有的测试套件犯了一个集成层面的致命错误：
- **单元测试造假**：在针对提取函数（如 `parseMultiGradeResponse`）编写单元测试时，测试用例传入的都是“干干净净、无需清洗”的理想字符串，直接跳过了前端预处理管线中的 `cleanJSON`。
- **孤岛效应**：每个函数分别有自己的 UT，但没人测试把文本**先传给 A，再传给 B** 会有什么结果。（L1 层极度冗余，L3 模块集成层为 0）。

### 5.2 解决方案：强制“穿透式”集成点测试 (Penetration Test)
AI 驱动生成的单元测试由于其固有的“Mock 偷懒”倾向，常常会写出完美避开系统瓶颈的隔离用例。架构师必须通过 PRD 强制纠偏：

1. **确立输入数据的保真度**：要求 AI 补充的测试用例中，`Input` 必须直接复制来自于物理真实大模型（如 SiliconFlow/NVIDIA）输出的**脏数据长串**（包含乱套的 `\begin{aligned}` 或畸形的外围符号），而不是人类手工精简过的 `[{"q": 1}]`。
2. **端到端管线验证**：除了孤立测试某个解析函数，必须强制要求写一个**调用总入口 (Handler/Service)** 的契约测试，模拟带有真实物理特征的脏字符串流动，验证前缀预处理和反序列化的整体验收。
3. **消除越权函数**：像 `cleanJSON` 产生越权清洗，是因为函数职责定义过宽。AI 时代防御这种问题的核心在于**防范面条式入口**，在设计层面就应将 `纯文本 OCR 清洗` 和 `JSON 强制提取` 切割为两个物理隔绝的通路。

## 6. E2E 断言"放水"：测试全绿不等于功能正常

> **背景痛点**：2026-02-27，多题识别功能在用户实际使用时发现"多题手风琴样式没出现"，但 E2E 测试一直全绿通过。

### 6.1 原因分析：断言阈值与业务预期脱节

审查 `e2e_web_test.js` 的 Scenario 2 发现 5 个漏洞：

| 漏洞 | 说明 |
|------|------|
| `#result-content` 存在性判断无效 | 该 div 写死在 HTML 中，永远存在，无论是否触发多题渲染 |
| API 拦截数量未参与判定 | `capturedResultsCount` 只打了 log，不影响 pass/fail |
| DOM 卡片阈值过低 | `renderedCount >= 1` 即通过，单题也能过关 |
| 摘要文本不校验数量 | 只检查"包含'识别完成'"，不验证题目数 |
| 单题降级路径放行 | AI 返回 1 道题时走 `renderResults()` 渲染 1 卡片，断言照样通过 |

### 6.2 修复方案

1. **API 层强校验**：`capturedResultsCount >= 2` 且参与 `overallPassed` 判定
2. **DOM 层强校验**：`renderedCount >= 2`，否则判 Fail
3. **摘要正则校验**：提取"共发现 N 道题目"中的 N，要求 N >= 2
4. **TDD_E2 冒烟前置**：只有多题有效渲染时才通过渲染质量检查

### 6.3 教训提炼

> **断言要和业务规格强绑定。** 测试通过的标准不是"不崩溃"，而是"业务预期被满足"。多题场景的通过标准必须是"至少渲染 2 道卡片"，而非"至少渲染 1 道"。否则测试 Pass 只是数字上的自欺欺人。

## 7. 如何保证 AI 生成断言的质量？

> **背景**：2026-02-27 对全量测试代码进行审查后发现，AI 生成的 Go 后端单元测试质量较高，但 E2E 断言存在系统性漏洞。本章基于审查经验提炼 AI 断言质量保障的方法论。

### 7.1 AI 断言的 5 大通病（从审查中总结）

| 通病 | 表现 | 真实案例 |
|------|------|---------|
| **"不崩即过"心态** | 断言只验证"不报错"而非"结果正确" | E2E 只检查 `renderedCount >= 1`，单题也能通过多题场景 |
| **阈值偷懒** | 用宽松阈值（`> 0`）替代精确阈值（`>= 2`） | API 拦截数量只打 log 不参与 pass/fail 判定 |
| **错误静默吞噬** | 丢弃 error 返回值，后续 panic 信息晦涩 | `json.Unmarshal()` 返回值被 `_` 丢弃 |
| **类型断言裸奔** | 直接 `.(map[string]interface{})` 无保护 | 接口结构变化时 panic 而非可读的断言失败 |
| **存在性代替正确性** | 检查"元素存在"而非"内容符合预期" | `#result-content` 是静态 HTML，永远存在 |

### 7.2 四道防御工事

#### 工事一：变异测试思维 (Mutation Testing Mindset)

审查每条断言时，问自己一个问题：

> **"如果我故意把被测代码改错（比如多题只返回 1 题），这条断言能抓住吗？"**

如果答案是"不能"，这条断言就是无效的。本次 E2E 审查的 5 个漏洞，全部可以用这一个问题检测出来。

#### 工事二：断言强度分级审查

| 强度 | 定义 | 示例 | 是否接受 |
|------|------|------|---------|
| **L0 存在性** | 检查元素是否存在 | `count > 0` | 仅用于冒烟测试 |
| **L1 非空** | 检查返回值非空 | `len(title) > 0` | 不可用于业务逻辑验证 |
| **L2 类型** | 检查类型正确 | `assert.IsType()` | 仅作为中间保护 |
| **L3 值域** | 检查值在业务范围内 | `count >= 2` | 最低可接受标准 |
| **L4 精确匹配** | 检查值完全等于预期 | `assert.Equal(t, 3, count)` | 推荐 |

**规则：业务关键路径的断言不得低于 L3。**

#### 工事三：反向 Prompt —— 让 AI 自审

在 AI 生成测试后，追加一轮 Prompt：

```
审查你刚才写的测试断言。对每条 assert，回答：
1. 如果被测代码返回错误结果但格式正确，这条断言能否抓住？
2. 断言的阈值是否和业务规格一致（而非"只要不崩就行"）？
3. 是否有 error 返回值被丢弃？
列出所有"不能"的断言并修复。
```

#### 工事四：人工审查红线清单

在 Code Review 中，以下模式直接打回：

- `json.Unmarshal(...)` 的 `error` 被 `_` 丢弃
- 类型断言 `.(Type)` 无 `ok` 检查或 `require` 保护
- E2E 断言阈值为 `>= 1` 或 `> 0` 用于多元素场景
- 只检查 HTTP 状态码不检查 Body 内容
- 用 `log` / `console.log` 代替 `assert` 做"伪断言"

### 7.3 为什么 Go 后端测试质量高于 E2E？

| 因素 | Go 后端 UT | E2E (JS/Python) |
|------|-----------|-----------------|
| **语言类型系统** | 编译期强类型，结构体字段不匹配直接报错 | 弱类型，运行时才暴露 |
| **AI 训练语料** | Go testing + testify 范例丰富且规范 | Playwright E2E 场景多样，AI 倾向保守 |
| **断言粒度** | 精确到字段值（`assert.Equal`） | 倾向于存在性检查（`count > 0`） |
| **反馈速度** | 编译不过就报错 | 必须启动浏览器运行才能发现 |

> **结论**：AI 在强类型语言 + 成熟测试框架场景下的断言质量较高；在弱类型 + E2E 场景下，需要人工更强力的审查和约束。

## 8. CD 部署"假成功"：三层 Bug 导致新代码从未上线

> **背景**：2026-02-27，用户反馈 APK 多题手风琴样式不生效。前端代码与本地完全一致，根因在远程服务器仍运行旧版后端。

### 8.1 Bug 链条（三层叠加）

| 层级 | Bug | 后果 |
|------|-----|------|
| **L1: systemd 路径不一致** | `ExecStart` 指向 `/root/studyInspire/study-inspire-backend`（旧路径），而 cicd.py 上传到 `/root/release/studyInspire-linux-amd64` | `systemctl restart` 重启的是旧二进制 |
| **L2: pkill 进程名不匹配** | `pkill -f studyInspire` 匹配不到旧进程名 `study-inspire-backend`（大小写和分隔符不同） | 手动重启路径也失败，旧进程占端口 |
| **L3: ldflags 未注入 Key** | `cicd.py` 的 `go build` 没有 `-ldflags`，API Key 未编译进二进制 | 即使新二进制成功启动，也因无 Key 而 500 |

### 8.2 修复方案

1. **systemd 指向正确路径** + 补充 `SILICONFLOW_API_KEY` 环境变量
2. **pkill 增加旧进程名匹配**：`pkill -f study-inspire`
3. **cicd.py 加 `-ldflags`**：从 `.env` 读取 Key 编译注入
4. **部署后健康检查**：cicd.py 自动验证 `/ping` 可达 + 前端 DOM 包含关键元素

### 8.3 教训

> **部署成功 ≠ 服务生效。** 每次部署必须有**端到端的健康检查**，至少验证：(1) 服务可达 (2) 前端资源完整 (3) 核心 API 功能正常。否则"部署成功"只是文件拷贝成功，不代表新代码在服务。
