# 测试经验总结：新项目测试开展指南

> 从 StudyInspire 项目实战中提炼，旨在指导新项目如何从零构建高质量、高弹性的测试体系。

---

## 1. 核心理念：金字塔防护与防御性思维

开展新项目测试的核心不在于“写多少代码”，而在于**“成本与置信度的平衡”**以及**“对不确定性的零信任假设”**。

### 1.1 分层架构
- **L1 单元测试**：逻辑的“原子围栏”，追求穷举与毫秒级反馈。
- **L3 契约测试**：数据的“法律依据”，拦截 AI 输出的波动。
- **L4 故障仿真**：系统的“压力测试”，主动注入故障验证降级。

---

## 2. 第一阶段：基准对齐与 SSOT 建设

在动工前，必须确立“单一事实来源”（Single Source of Truth）。

1.  **文档锚定**：在代码实现前，将降级顺位（如 Provider 顺序）、重试策略写入 PRD。文档是测试的“法理依据”。
2.  **环境隔离矩阵**：
    *   **本地执行**：禁止 Mock，验证真实链路。
    *   **CI 执行**：允许协议仿真（Mocking），追求流水线通过率与速度。

---

## 3. 第二阶段：原子级防御（UT 穷举美学）

利用 L1 层的稳定性，构建最底层的“安全网”。

1.  **纯函数穷举**：针对文本清洗（LaTeX/JSON 提取）逻辑，覆盖 100% 的分支。
2.  **协议仿真 (httptest)**：劫持传输层而非简单 Mock 接口。强迫生产代码处理真实的 HTTP Response（包括报头与错误码）。
3.  **强制性 Error Path**：每个 API 调用必须包含：
    *   **Happy Path**: 200 OK。
    *   **Logic Error**: 4xx/5xx 处理。
    *   **Transport Error**: 模拟网络断开（返回 `nil`），杜绝空指针 Panic。

---

## 4. 第三阶段：契约化防御（抗脆性治理）

针对 AI 输出的不确定性，建立强校验门禁。

1.  **Schema 强校验**：使用 `jsonschema` 校验 API 物理结构、字段类型及业务约束（如禁止特定的 LaTeX 符号）。
2.  **解耦“提纯”与“清洗”**：
    *   **提纯层**：精准定位 JSON 闭合括号，剥离 Markdown 废话。
    *   **清洗层**：Unmarshal 成功后，再进行业务文本修正（Sanitize）。

---

## 5. 第四阶段：主动防御（故障仿真）

变“被动感知”为“主动扎针”。

1.  **故障注入测试 (Fault Injection)**：模拟主力服务持续 401/503/超时。
2.  **降级链路锚定**：验证系统在主力挂掉后，是否能按预期顺位秒级切换到备份节点。
3.  **弹性参数验证**：测试重试退避（Backoff）算法，防止请求堆积引发雪崩。

---

## 6. 第五阶段：工程化闭环（治理与复用）

1.  **Test Utilities 工具层**：封装统一的 `APIClient`。禁止在各处硬编码 Key 或 URL。
2.  **报告聚合**：跨语言测试（Python/Node/Go）必须产出统一 JSON 格式，挂载到 CI 进行中心化度量。
3.  **三级质量门禁**：
    *   **Pre-push**: 强制跑完 L1 单元测试。
    *   **PR Check**: 自动执行回归与契约测试。
    *   **Post-deploy**: 生产环境健康检查（/ping）。

---

## 7. 投入产出比 (ROI) 与工程实践分界线

在饱和式测试投入后，需审视哪些投入是“护城河”，哪些是“过度工程化”。

### 7.1 ROI 定量分析
- **Go 后端 UT (核心逻辑)**: **极高 ROI**。拦截了诸如 RequestID 传递失效、Context 取消不敏感等隐性 Bug。它是将“运行时猜测”转为“编译后契约”的刚需投入。
- **Python 测试框架 UT (工具链自验)**: **高 ROI (可信度溢价)**。确保了“尺子本身是准的”。避免在集成测试失败时，开发者去怀疑是测试脚本本身有 Bug。

### 7.2 实践推广边界 (The Boundary)
*   **必须推广的领域**：跨 3 个以上模块引用的基础构件（如网络客户端、日志库、配置加载器）。地基代码必须 100% 覆盖 Happy/Failure Path。
*   **建议“以测代证”的领域**：纯业务性质的回归脚本。脚本本身即是测试，无需为测试脚本补写过多的单元测试，除非逻辑极度复杂。

### 7.3 “三不测”原则 (Goldilocks Zone)
1.  **琐碎逻辑不测**：纯 Setter/Getter、简单常量定义及标准库原生行为。
2.  **第三方框架库底层不测**：不应测试 Gin 或 Requests 库内部是否工作。
3.  **UI 瞬时中间态不测**：除非涉及复杂算法，否则交互状态应交给高层的 E2E。

### 7.4 AI 原生开发的特殊“度”
凡涉及 **非结构化数据处理**（正则表达式提取、JSON 容错清洗、数学符号转换）的函数，必须实施**像素级 UT 覆盖**。AI 输出具有概率性，因此解析端的代码必须具备 100% 的确定性。

## 8. CI/CD 防漂移机制：让本地与流水线永远同步

### 8.1 核心问题：两处"事实来源"必然产生漂移

当业务代码的 Provider 列表（如 OCR/Grading 厂商）分散维护在多处时，漂移是必然的，而非偶然的：

```
run_tests.py  ← 本地测试知道用哪些 Provider
ci-cd.yml     ← CI 流水线独立维护 secrets 注入列表
```

典型症状：本地测试正常，CI 因 API Key 未注入而静默失败（不报错，只是用空 Key 绕过）。

### 8.2 解决方案：SSOT（单一事实来源） + Pre-commit 守护

**原则：CI yaml 只做外交官（搬运 Secrets），业务判断只在测试脚本里。**

1. **在 `run_tests.py` 中声明 `PROVIDERS_IN_USE` 常量**（唯一事实来源）：

```python
PROVIDERS_IN_USE = [
    {"key": "SILICONFLOW_API_KEY", "role": "OCR + 判卷 (首选)", "tier": "primary"},
    {"key": "NVIDIA_API_KEY",      "role": "Fallback 判卷",      "tier": "fallback"},
    {"key": "ZHIPU_API_KEY",       "role": "Fallback OCR",       "tier": "fallback"},
]
```

2. **CI yaml 只做通用的 Secret 传递**，不写任何业务判断：

```yaml
env:
  SILICONFLOW_API_KEY: ${{ secrets.SILICONFLOW_API_KEY }}
  NVIDIA_API_KEY:      ${{ secrets.NVIDIA_API_KEY }}
  ZHIPU_API_KEY:       ${{ secrets.ZHIPU_API_KEY }}
```

3. **在 `pre_commit_check.py` 里增加 `check_ci_provider_sync()`**：在提交时自动读取 `PROVIDERS_IN_USE`，与 `ci-cd.yml` 中的 `secrets.*` 做比对，一旦发现漂移立即阻断提交。

### 8.3 效果

- **新增 Provider 流程**：①改 `PROVIDERS_IN_USE` → ②`git commit` 时守护脚本自动提示 CI yaml 需要补充 → ③补上对应 Secret。
- **删除 Provider 流程**：①从 `PROVIDERS_IN_USE` 移除 → ②推送后 CI yaml 留有未使用的 Secret 注入（无害），或一起清理。
- **永远不会漂移**：因为提交钩子在本地就拦截了不一致，根本到不了 CI。

### 8.4 教训提炼

> **凡是"同一个知识"需要在多于一处维护，就应该引入 SSOT + 自动化守护。** 靠人工记得同步是不可靠的。测试基础设施本身的一致性与业务代码一样重要，不应该"靠约定"来维护。

## 9. 日志可见性平衡：静默执行与错误摘要 (Error Summary)

### 9.1 痛点：被吞没的错误与无效的翻阅
在复杂的自动化管线中，我们常陷入两个极端：
- **全量日志模式**：数千行成功的输出淹没了真正的错误原因，开发者被迫在控制台拼命向上翻找。
- **静默模式**：只显示 `[FAIL]`，不给任何提示，迫使开发者重新带 `-v` 手动跑一遍复现。

### 9.2 解决方案：捕获模式 + 尾部摘要
本项目在 `run_tests.py` 中引入了“双重性格”的 `run_cmd`：
1. **执行时静默**：通过 `capture_output=True` 捕获所有 `stdout/stderr`，控制台只显示整洁的进度。
2. **失败时爆破**：一旦 ExitCode 不为 0，将捕获到的输出存入 `TestCase.output`。
3. **底部汇总汇报**：在所有测试结束后，单辟一个 `Error Summary` 区域，统一打印所有失败阶段的 **Last 30 lines**。

### 9.3 核心教训
**测试失败的第一秒，开发者就应该看到 Root Cause。** 好的测试脚本不应该让用户“去查日志”，而应该主动“递上现场”。

## 10. 前端轻量化自验：Jest + JSDOM 的降维打击

### 10.1 场景错位
以往我们习惯用 Playwright/Cypress 测前端，但它们的开销极大（启动浏览器 > 5s）。
- **痛点**：只是改了一个 DOM ID 或简单的模板渲染逻辑，却要等 30 秒的浏览器冷启动和网络等待。
- **方案**：引入基于 Node.js 的 **Jest + JSDOM**。

### 10.2 降维打击的优势
1. **秒级反馈**：6 个核心业务用例运行仅需不到 1 秒。
2. **异常模拟**：可以在单元测试中轻易 mock `global.fetch` 分别返回 500、401 或模拟网络超时（Abort），这在真实的 E2E 中极难实现且不稳定。
3. **及早拦截**：将其放在 Phase 1.6（L1层），如果前端基础渲染都挂了，就没必要启动后端的 `go build` 和漫长的 E2E 测试。

### 10.3 注意事项：环境偏差
- **InnerText 不可靠**：JSDOM 对 `innerText` (依赖布局渲染) 的支持较差，断言建议统一使用 `textContent`。
- **全局变量暴露**：对于直接运行在浏览器的脚本 (`app.js`)，需要显式将核心函数挂载到 `window` 才能在 Node 测试环境调到。
