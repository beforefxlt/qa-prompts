## 二、变量命名规范

为避免混淆，在 Dify/Coze 或内部平台统一变量命名：

| 变量名 | 含义 | 示例 |
| :--- | :--- | :--- |
| `{{project_name}}` | 项目名称 | 退休返聘系统 |
| `{{module_name}}` | 模块名称 | 订单管理模块 |
| `{{field_name}}` | 字段名称 | 用户年龄 |
| `{{data_type}}` | 数据类型 | int, string, date |
| `{{constraints}}` | 约束条件 | 18-65, 长度 5-20 |
| `{{business_logic}}` | 业务逻辑说明 | 退休返聘，非此区间无法注册 |
| `{{requirement_text}}` | 需求原文 | PRD/用户故事完整文本 |
| `{{api_spec}}` | API 规范 | Swagger/JSON 格式 |
| `{{business_rules}}` | 业务规则列表 | Token 有效期、积分规则等 |
| `{{testing_goal}}` | 测试目标 | 探索性测试的 charter |
| `{{state_list}}` | 状态列表 | 待支付，已支付，已完成 |
| `{{action_list}}` | 动作列表 | 用户支付，系统超时 |

## 三、Few-Shot 嵌入规范

每个模板末尾必须包含 1-2 个"优质范例"。

**原因**：LLM 对"例子"的学习效果远好于对"指令"的理解。

### 范例结构

```markdown
## 使用示例

### 示例 X：[场景名称]

**输入变量**：
{清晰列出所有变量值}

**预期输出**：
{完整的 AI 生成结果}
```

### 范例质量要求

- ✅ 输入变量真实、具体（避免占位符）
- ✅ 输出覆盖所有约束条件
- ✅ 包含边界情况和异常场景
- ✅ 输出格式可直接使用

## 四、动态反馈循环

### 问题记录流程

```
测试人员发现 AI 生成问题
        ↓
记录到 Issues（标签：prompt-quality）
        ↓
QA 架构师分析根因
        ↓
   ┌────┴────┐
   ↓         ↓
模板约束不足  AI 能力限制
   ↓         ↓
更新模板    记录 workaround
   ↓
发布新版本
```

```markdown

{应该包含 min-1 场景}

**建议修复**:
在 Constraints 中明确强调必须包含 min-1
```

## 五、IDE 集成方案

### VS Code Snippet 配置

在 `.vscode/snippets.json` 中配置：

```json
{
  "P2P4 极值注入分析": {
    "prefix": "p2p4",
    "body": [
      "# Role: 高级软件测试架构师",
      "",
      "# Context:",
      "当前正在对 [${1:项目名称}] 执行 **P2 (数据异常)** 与 **P4 (边界条件)** 维度的防御性测试设计。",
      "",
      "# Input Data:",
      "- 目标字段：${2:字段名}",
      "- 数据类型：${3:int}",
      "- 约束范围：${4:1-100}",
      "- 业务逻辑备注：${5}",
      "",
      "# Task:",
      "请根据上述输入，输出一套标准的边界值测试矩阵。",
      "",
      "# Constraints:",
      "1. 必须覆盖：最小值 (min)、最小值减 1(min-1)、最小值加 1(min+1)",
      "2. 必须覆盖：最大值 (max)、最大值加 1(max+1)、最大值减 1(max-1)",
      "3. 包含非法值检测：空值、特殊字符、超长字符串",
      "4. 输出格式必须为 Markdown 表格"
    ],
    "description": "生成边界值分析测试用例"
  }
}
```

### 使用方式

1. 在 Markdown 文件输入 `bva`
2. 按 Tab 键展开模板
3. 依次填写变量

## 七、模板开发 Checklist

开发新模板时，请确保：

- [ ] 模板结构符合 RCTF 架构（Role-Context-Task-Constraint）
- [ ] 变量定义清晰，符合命名规范
- [ ] 约束条件完整，无歧义
- [ ] 输出格式标准化（表格/JSON/Mermaid）
- [ ] 至少包含 2 个实际示例（输入 + 输出）
- [ ] 示例覆盖正常场景 + 异常场景
- [ ] 包含变更记录表格
- [ ] 通过团队评审
