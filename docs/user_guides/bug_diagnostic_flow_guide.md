# 📘 指南：缺陷初诊与隔离流水线 (`bug-diagnostic-flow`)

本指南介绍如何利用 `bug-diagnostic-flow`，将现场的模糊缺陷现象逐步收敛为**可追踪的根因隔离结论**与**标准化 `BUG-001` 草稿**。

---

## 1. 什么是 `bug-diagnostic-flow`？

`bug-diagnostic-flow` 不是直接“帮你写问题单”的快捷提示词，而是一套三段式缺陷隔离 SOP：

1. **问诊补证**：先补齐缺陷发生时的关键上下文、日志、截图与环境信息。
2. **根因隔离**：基于 `/tmp/` 中的物理证据做 D1-D5 视角推演（Dimension），形成可复核的初步结论。
3. **严重程度定级**：依据决策树对缺陷进行自动定级 (P0/P1/P2)。
4. **草稿建单**：复用 `issue-reporter` 生成 `BUG-001` 草稿。

该流程的目标不是追求“立刻写单”，而是避免把模糊现象直接升级成低质量问题单。

---

## 2. 如何触发？

在与 AI 对话中输入以下指令（Slash Command）：

```bash
/bug-diagnostic-flow [缺陷现象描述、日志摘要或工单内容]
```

例如：

```bash
/bug-diagnostic-flow 通讯偶发中断，现场反馈重启后暂时恢复，但 30 分钟左右又出现。
```

---

## 3. 标准操作流程 (3 步流)

### 第一步：症状初诊与证据收集

AI 会先把当前信息当作“问诊素材”，而不是直接写正式单子。

- **用户操作**：补充 AI 追问的关键缺失项，例如日志路径、截图、运行状态、硬件版本、网络状态。
- **中间产物**：`/tmp/bug_investigation_raw.md`
- **关键指令**：回复 **“开始推演”**。

### 第二步：根因推演与隔离分析

AI 只基于 `/tmp/bug_investigation_raw.md` 的事实信息继续分析，避免被上游对话噪音带偏。

- **分析视角**：
  - 是底层异常还是业务拦截
  - 更接近 D3 环境/基建问题，还是 D4 边界触发，或 D5 交付与使用问题
  - 哪些模块、组件或链路最值得优先排查
- **中间产物**：`/tmp/bug_root_cause_analysis.md`
- **关键指令**：回复 **“建单”**。

### 第二.五步：严重程度定级

AI 调用 `bug-triage-rules` 对识别出的根因进行严重程度评估。

- **定级标准**：涉及安全/保护功能失败直接定级为 P0。
- **关键动作**：向用户展示定级结论及命中因子，确认无误后进入建单。

### 第三步：标准缺陷草稿生成

AI 会优先调用 `issue-reporter`，按 `templates/defect-report-template.md` 生成 `BUG-001` 草稿。

- **中间产物**：`/tmp/bug_report_draft.md`
- **输出内容**：
  - 标准化缺陷草稿
  - 根因隔离与定级结论说明

---

## 4. 资产边界

这是本流程最重要的约束之一：

1. `BUG-001` 草稿默认属于**单次诊断产物**
2. 草稿默认只落在 `/tmp/bug_report_draft.md`
3. 默认**不直接进入 Git 仓库**
4. 若要继续同步，应优先进入外部 Issue 系统或用户明确指定的非仓库交付路径

这条边界是为了避免把一次性分析过程误沉淀为正式仓库资产。

## 5. 实战案例

### A. 初始输入

> **User**: `/bug-diagnostic-flow EMS 页面偶发显示设备离线，但后台 104 链路监控没有持续断开。`

### B. AI 问诊

> **AI**: “请补充以下信息：  
> 1. 发生时的日志路径  
> 2. 是否发生过主备切换  
> 3. 设备型号与版本号  
> 4. 页面离线持续时间与恢复方式”

> **User**: “日志在 `/var/log/ems/app.log`，发生前 2 分钟有主备切换，版本 V2.3.4，页面离线约 10 秒后恢复。”

> **AI**: “初始病情已固化至 `/tmp/bug_investigation_raw.md`。请回复‘开始推演’。”

### C. AI 隔离

> **AI**: “初步判断更接近交互层状态同步问题，而不是持续链路中断。已生成 `/tmp/bug_root_cause_analysis.md`。如同意按该方向建单，请回复‘建单’。”

### D. AI 生成草稿

> **AI**: “`BUG-001` 草稿已生成并固化至 `/tmp/bug_report_draft.md`。已同步包含 P0 级定级因子说明。”

---

## 6. 为什么这套流程更稳？

1. **先补证，再分析**：避免把口头现象直接写成问题单。
2. **阶段落盘**：每一步都把事实和结论固化到 `/tmp/`，降低上下文漂移。
3. **分析与建单分离**：隔离结论成熟后再生成标准草稿。
4. **资产边界清晰**：缺陷分析草稿默认不入库，符合仓库治理规则。
