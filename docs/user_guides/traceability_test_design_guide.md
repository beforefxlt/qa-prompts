# 可追溯测试架构设计指南

> 版本：v1.0.0  
> 更新日期：2026-04-01  
> 适用对象：测试架构设计者、需求评审者、负责测试落地的 AI Agent

## 1. 这份指南解决什么问题

当项目规模变大后，测试最容易出现三类失真：

1. 规格里写了很多能力，但代码和测试没有形成闭环。
2. 测试文件很多，但无法回答“这一条测试到底覆盖了哪条需求”。
3. 测试跑完只剩 `Pass/Fail`，无法给出“总规格数、自动覆盖数、缺失数、失配数”。

本指南的目标不是教人“多写测试”，而是建立一套可以长期复用的测试设计框架，让团队始终回答清楚下面四个问题：

1. 我们到底承诺了哪些测试条目。
2. 每一条条目映射到哪些代码和哪些测试。
3. 哪些条目已经自动化覆盖，哪些还没有。
4. 当规格、测试、实现发生漂移时，谁来发现，怎么阻断。

## 2. 核心原则

### 2.1 规格先于测试编号

测试编号不是为了装饰报告，而是为了把“规格语言”转换成“工程语言”。

推荐做法：

- 先从 `PRD.md`、`UI_SPEC.md`、`API_CONTRACT.md`、`TEST_STRATEGY.md` 等文档拆出正式测试条目。
- 再为条目分配稳定编号，例如 `TC-P1-001`、`TC-P5-014`。
- 任何测试代码都引用编号，而不是重新发明自然语言标题。

### 2.2 Traceability 是真理之源，不是附属表格

必须维护一个机读映射文件，例如 `traceability.yaml`，作为全项目测试可追溯关系的唯一真理之源。

它至少应该回答：

- `tc_id` 是什么。
- 规格来源是什么。
- 覆盖层级是什么。
- 对应的测试文件和测试函数是什么。
- 关联代码文件是什么。
- 当前状态是 `automated`、`stub`、`manual` 还是 `missing`。

如果没有这层映射，任何“覆盖率”都只是目录感知，不是真正的规格覆盖率。

### 2.3 设计顺序必须是 Design -> Test Stub -> Code

对功能性规格变更，推荐强制采用以下顺序：

1. 先改规格文档。
2. 同步新增或修改对应 TC 条目。
3. 先提交测试 Stub，且应处于失败态或待实现态。
4. 再补实现代码。
5. 最后补齐通过态断言与审计报告。

这样做的目的不是形式化，而是防止“代码先漂移，规格和测试事后追认”。

### 2.4 测试分层必须物理落地

测试分层不能只写在文档里，必须反映到目录结构上。推荐至少划分为：

- `unit/`
- `contract/`
- `integration/`
- `resilience/`
- `golden/`

物理分层的价值：

- 让执行入口可以分层运行。
- 让审计脚本能按层统计。
- 让评审者快速判断一条能力落在哪个测试面。

### 2.5 审计输出必须面向“缺口”，不是只面向“通过率”

真正有用的测试汇报不应只说“本轮通过 95%”，而应回答：

- 总条目数
- 自动覆盖数
- 缺失数
- 失配数
- Stub 数
- 每层级覆盖分布

示例：

```text
总 TC：107
自动覆盖：72
测试 Stub：8
缺失：27
失配：5
```

这类输出才有资格支撑质量门禁。

## 3. P1-P5 分层模型

这套框架推荐将测试意图先映射到 P1-P5，再映射到执行层。

### P1：主流程成功路径

回答“系统最核心的价值流能否完成”。

常见对象：

- 核心 CRUD 主路径
- 关键提交链路
- 用户最常走的业务闭环

### P2：领域错误与业务规则失败

回答“错误输入、非法状态、规则冲突是否被正确拒绝或降级处理”。

常见对象：

- 非法枚举
- 缺字段
- 规则冲突
- 审批条件不满足

### P3：基础设施与外部依赖异常

回答“数据库、对象存储、消息队列、网络、第三方服务异常时，系统是否还能给出正确反馈并保护数据一致性”。

常见对象：

- 存储写失败
- 外部接口超时
- 服务不可达
- 重试与补偿

### P4：边界、阈值与时间窗口

回答“临界条件是否定义清楚，系统在边缘值附近是否稳定”。

常见对象：

- 数值上下限
- 日期边界
- 时间窗切换
- 列表空态和单元素态

### P5：交付视角与用户体验验收

回答“从交付者和使用者视角看，这个系统是否真的可用、可理解、可恢复、可运维”。

常见对象：

- 空状态是否可理解
- 错误提示是否明确
- 引导路径是否闭环
- 页面信息是否完整

## 4. 规格、测试与代码的三层映射

推荐稳定采用如下链路：

```text
SPEC 条目 -> TC 编号 -> traceability.yaml -> 测试函数 -> 代码文件
```

其中每一层都承担不同职责：

- `SPEC` 定义承诺。
- `TC` 定义最小验收单元。
- `traceability.yaml` 定义映射关系。
- 测试函数定义验证动作。
- 代码文件定义最终实现。

只要其中一层缺失，就会出现“文档说有、测试说有、代码其实没有”或者反过来的悬空状态。

## 5. 推荐目录模型

```text
backend/tests/
  unit/
  contract/
  integration/
  resilience/
  golden/

frontend/e2e/
  *.spec.ts

traceability.yaml
scripts/
  qa_all.py
  verify_traceability.py
  verify_contract.py
```

建议约定：

- 后端测试函数名可以保持技术语义。
- 前端 E2E 的 `test()` 描述必须显式带 `TC-P5-xxx`。
- 审计脚本统一从 `traceability.yaml` 向下解析，而不是反过来猜测。

## 6. traceability.yaml 的最小字段建议

推荐每条记录包含：

```yaml
- tc_id: TC-P5-001
  title: 首页空状态文案清晰可读
  level: p5
  layer: e2e
  spec_source:
    - docs/specs/UI_SPEC.md
    - docs/specs/test_strategy_matrix.md
  code_paths:
    - frontend/src/app/page.tsx
    - frontend/src/components/forms/MemberForm.tsx
  tests:
    - file: frontend/e2e/ux-experience.spec.ts
      name: "TC-P5-001 首页空状态文案清晰可读"
  status: automated
```

额外可选字段：

- `owner`
- `risk`
- `notes`
- `stub_only`
- `manual_evidence`

## 7. 统一执行入口应该做什么

建议统一入口脚本做三件事，并优先使用 Python，而不是 PowerShell。

1. 分层执行自动化测试。
2. 执行规格和契约扫描。
3. 输出统一审计摘要。

推荐顺序：

1. `python scripts/qa_unit.py` 或等价入口执行后端轻量验证
2. `python scripts/qa_contract.py`、`python scripts/qa_integration.py` 等执行分层测试
3. `python scripts/qa_e2e.py` 执行前端正式 E2E
4. `python verify_contract.py` 或同类脚本执行契约对齐检查
5. `python verify_traceability.py` 输出覆盖审计结果

统一入口的职责不是“方便一键运行”，而是固定质量口径。

### 7.1 分层编排脚本应在项目初始化时就创建

不要等测试很多了再补执行脚本。

推荐做法是：在项目刚建立测试骨架时，就同时创建分层编排脚本，即使当下每一层只有 0 到 1 个测试也要先建空壳。

推荐最小集合：

```text
scripts/
  qa_unit.py
  qa_contract.py
  qa_integration.py
  qa_resilience.py
  qa_golden.py
  qa_e2e.py
  qa_all.py
  qa_audit.py
```

建议职责：

- `qa_unit.py`：仅跑最轻量、最快速的单元验证
- `qa_contract.py`：跑 API 契约与规格对齐
- `qa_integration.py`：跑后端协作链路
- `qa_resilience.py`：跑异常与容灾验证
- `qa_golden.py`：跑黄金样本与基线对比
- `qa_e2e.py`：跑前端正式验收流
- `qa_all.py`：跑交付态全链执行
- `qa_audit.py`：只做追溯统计、缺口统计和门禁摘要

这样做有三个直接收益：

1. 后续新增测试时有固定挂载点，不会到处散落。
2. 本地开发、CI 和交付审计从第一天起就使用同一套口径。
3. 随着用例增长，只需要往既有集合里追加，不需要中途再重构执行方式。

推荐原则：

- 先创建脚本骨架，再逐步把测试挂进去。
- 脚本名保持稳定，不随项目阶段频繁改名。
- 默认假设未来会进入 Linux/macOS CI/CD，因此不要把 `.ps1` 作为主入口。
- `traceability.yaml` 应记录条目所属的执行层，便于脚本和审计统计联动。

## 8. 机读门禁应该拦什么

### Step 4：可追溯性门禁

至少检查：

- `traceability.yaml` 中新增条目是否存在对应测试函数。
- 测试函数引用的文件是否真实存在。
- 新增测试是否带合法 `TC-*` 编号。
- 新增规格条目是否同步进入追溯映射。

### Step 7：TDD 顺序门禁

至少检查：

- 修改 `SPEC.md` 时是否同步修改了测试或新增了测试 Stub。
- 规格新增但没有测试条目时，阻断提交。
- 出现新页面或新 API 规格，但代码没有实现或没有 stub 时，阻断提交。

注意：纯文案修订、错别字修正、非功能性说明调整，可以降低门禁等级，避免误伤。

## 9. 常见反模式

### 9.1 目录分层了，但没有条目映射

问题：

- 看上去“很工程化”，实际上无法回答规格覆盖率。

### 9.2 测试名字写得很好，但没有编号

问题：

- 后期很难自动统计缺口，也无法和规格保持稳定绑定。

### 9.3 先写代码，再补测试编号和规格

问题：

- 会把测试降级为“事后证明”，不是“设计约束”。

### 9.4 用目录数量冒充覆盖率

问题：

- 文件多不代表覆盖全；一个文件里可能只覆盖一个主路径。

### 9.5 只报通过率，不报缺口

问题：

- 通过率可能很高，但缺失条目同样很多，管理上会被严重误导。

## 10. 设计新项目时的推荐动作

当你要为一个新项目设计测试体系时，建议按这个顺序推进：

1. 先冻结规格来源文档。
2. 把规格拆成正式 TC 列表。
3. 先建 `traceability.yaml` 初稿。
4. 先放测试 Stub，再分层实现。
5. 建统一执行入口。
6. 建覆盖审计脚本。
7. 把门禁接进工作流。

## 11. 判断这套架构是否落地成功

满足以下条件时，才算真正落地：

1. 任意一条 `TC-Px-yyy` 都能定位到规格、测试函数、代码文件。
2. 任意一次规格变更都会触发追溯检查。
3. 统一入口能输出总条目、自动覆盖、缺失和失配统计。
4. 新增功能默认先有 stub，再有实现。
5. 评审者不需要靠聊天上下文，也能读懂当前测试覆盖状态。

## 12. 建议与 Skill 的关系

这份指南负责说明“为什么这样设计”。  
对应的 Skill 负责说明“下次如何按这套方法执行”。

推荐把两者同时维护：

- 指南沉淀方法论。
- Skill 固化执行动作。

只有这样，团队才不会在下一个项目里又从零讨论一遍。
