# QA 收口与简化通用原则

本文档沉淀 `qa-prompts` 中与“收口复核”“测试代码简化”相关的共性原则，供工作流、Skill 与用户指南复用。

## 1. 适用边界

本原则适用于以下场景：
- 对 QA 工作流产物做最终一致性复核。
- 对测试代码做低扰动简化与整理。
- 在提交前判断哪些文件属于正式资产，哪些属于临时运行产物。

不适用于：
- 需求尚未稳定时的大幅改写。
- 业务实现代码的通用重构。
- 为了“更优雅”而扩大改动范围的整理。

## 2. 核心原则

1. **低扰动优先**：优先做小步、可验证、收益明确的收口动作。
2. **行为不变**：不能改变测试目标、测试覆盖面、失败语义与环境前提。
3. **信号优先**：整理后必须更容易定位问题，而不是更难读。
4. **文档同步**：工作流、README、用户指南、Skill 说明的口径必须一致。
5. **正式资产边界清晰**：临时工具输出、调试索引、仓库快照等不得混入正式资产或提交范围。

## 3. 临时产物判定

以下内容默认视为临时产物，而非正式资产：
- `.repomix-output.txt`
- 调试导出的仓库快照
- 截图索引、临时日志、抓包文件
- 运行过程中生成的 `__pycache__`、本地日志文件

处理原则：
- 不纳入文档资产清单。
- 不作为工作流最终交付物。
- 不进入提交范围，必要时加入 `.gitignore`。

## 4. Reviewer Agent 与 Test Code Simplifier 的分工

### Reviewer Agent
- 面向 QA 资产与工作流收口。
- 重点检查工作流编排、测试资产完整性、文档同步性、技能治理合规性。
- 不替代测试设计或代码实现。

### Test Code Simplifier
- 面向测试代码本身的低扰动整理。
- 重点收紧重复断言、fixture 职责、命名与失败信号表达。
- 不处理业务代码，也不是默认每轮都要调用。

## 5. 推荐使用顺序

1. 先完成需求审计、策略规划、测试产出或测试代码开发。
2. 若测试代码存在表达冗余，再按需调用 `test-code-simplifier`。
3. 在正式提交或交付前，再调用 `reviewer-agent` 做最终收口。

这样可以避免把“代码整理”和“资产复核”混成同一个动作。
