# 工业协议异常与鲁棒性测试演进调研

## 1. 当前套件已覆盖能力
*   **L1 (Parsing)**: 报文截断 (Truncated)、数值边界溢出 (Honeypot)。
*   **L2 (Interaction)**: 事务 ID 篡改 (Mismatch)、请求/响应频率不匹配 (Soak)。
*   **L3 (Link)**: 延迟注入 (Delay)、连接强制中断 (Drop)、并发连接上限 (MaxConns)。

## 2. 工业场景中常见但尚未覆盖的异常

### A. 协议一致性与畸形报文 (L1 进阶)
*   **超长报文攻击 (Oversized Payload)**: Modbus 标准上限是 260 字节。如果仿真器返回一个 500 字节甚至更大的 PDU，测试主站 (Master) 是否会发生缓冲区溢出或内存写越界。
*   **非法功能码 (Invalid OpCodes)**: 注入 0x2B (读取设备识别) 以外的非标准功能码（如 0x64/0x65）。受测程序是否能正确识别并抛出 Exception Code 01。
*   **MBAP 头部篡改**: 设置 Protocol ID 非 0（标准要求必须为 0），或者 Length 字段与实际 Payload 长度严重不符（但不是截断，而是冗余）。

### B. 业务逻辑与时序安全 (L2 进阶)
*   **重放攻击 (Replay Attack)**: 捕获正常的控制指令（如 6001 开机），在异常时刻重新发送。这在无线或非加密的 RS485/以太网中极具破坏性。
*   **TID 回绕与碰撞**: 在超高频（QPS > 1000）场景下，快速用尽 16 位 TID 通信空间，验证系统是否会出现数据关联错位。
*   **多从站 Unit ID 冲突**: 模拟一个网关挂载多个从站，并返回错误的 Unit ID 响应，测试主站的解析隔离性。

### C. 网络负载与链路压力 (L3 进阶)
*   **连接风暴 (Connection Flood)**: 短时间内发起数千个 TCP 连接请求。验证设备是否会因为文件句柄 (FD) 耗尽而导致内核 Panic。
*   **半开连接攻击 (SYN Flood)**: 模拟只建立 TCP 三次握手的前两次，长期占用从站的监听队列。
*   **分片重组异常 (MTU Fragmentation)**: 故将一个 Modbus 报文拆分成多个微小的 TCP 包发送（每包 1 字节），测试受测程序的粘包处理能力（这是很多嵌入式协议栈的死穴）。

## 3. 跨协议扩展建议 (CAN / IEC 61850)
*   **CAN 总线**: 模拟 **Bus-Off** 状态恢复、标识符优先级碰撞（ID 冲突）、高负载下的仲裁失败风险。
*   **IEC 61850 (GOOSE)**: 模拟 **StNum (状态号)** 跳变、**Time Allowed to Live** 超时、多播风暴下的 CPU 占用率。

## 4. 讨论建议
*   **优先级 1**: 实现 **Oversized Payload** (L1) 和 **TCP 分片/粘包模拟** (L3)。这对嵌入式设备的稳定性提升立竿见影。
*   **优先级 2**: 引入 **非标准功能码注入**，提高协议栈的标准化兼容性。
