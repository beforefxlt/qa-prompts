# 协议测试深层思考与实践感悟 (QA Insights)

## 1. 协议异常的分层演进模型 (Layered Anomaly Model)

在今日对 Modbus 靶机的改进中，我们跳出了单纯的“畸形包发包”思路，构建了更符合工业实战的四层异常模型：

*   **L1: 传输/环境异常 (Infrastructure Layer)**:
    *   *关注点*: Socket 句柄耗尽 (BUG_LEAK)、解析器死锁 (BUG_STACK)。
    *   *感悟*: 协议栈底层如果不处理好 `asyncio` 的超时或非法长度读取，系统会从物理层面上直接瘫痪。
*   **L2: 规约合规性异常 (Compliance Layer)**:
    *   *关注点*: 事务 ID 篡改 (BUG_MISMATCH)、非标长度。
    *   *感悟*: TID 匹配是协议驱动的基础，如果客户端不校验 TID，可能导致指令错乱（如将电池 A 的电量读数错配给电池 B）。
*   **L3: 业务逻辑异常 (Business Logic Layer)**:
    *   *关注点*: 关键地址数值溢出 (BUG_HONEYPOT)。
    *   *感悟*: 协议测试不能脱离业务。在储能场景下，读取到 `0xFFFF` 的功率值可能导致上层能量调度算法失效，这是比“程序崩溃”更隐蔽、更危险的逻辑 Bug。
*   **L4: 复杂时序异常 (Timing & Jitter Layer)**:
    *   *关注点*: 响应随机延迟 (BUG_DELAY)。
    *   *感悟*: 实验室环境往往太“干净”。真实工场环境的网络抖动（Jitter）是测试状态机（FSM）稳健性的唯一手段。

## 2. 靶机设计的“故意有缺陷”原则 (Deliberately Vulnerable)

靶机 (Target) 的价值不在于模拟正常的设备，而在于它能**稳定地产生可控的故障**。
*   如果一套测试框架（Fuzzer）无法在靶机上发现我们明着埋下的 `OOB` 或 `LEAK`，那么这套框架在面对真实设备时必然是“盲目”的。
*   靶机是测试工具的“磨刀石”。

## 3. 从“随机碰撞”到“意图驱动”的测试转变

传统的模糊测试 (Fuzzing) 盲目追求代码覆盖率。在协议测试中，我们更倾向于**场景堆叠攻击 (Scenario Stacking)**：
*   **示例**: `超长报文` + `TCP 分片发送` + `响应延迟`。
*   **实战价值**: 这种组合拳能极大压缩协议栈的重组空间，强迫隐藏在缓冲区管理逻辑中的 Race Condition（竞争条件）显现。

## 4. 未来展望：协议感知与上下文注入

未来的测试方向应不仅是发送“随机垃圾”，而是发送“针对性的业务垃圾”。
*   根据设备 Profile 自动生成“地址级”的蜜罐。
*   不仅测试响应，更应测试在连续报错后设备的“自愈能力”。
