# Modbus 异常测试框架设计与实现指南

## 1. 背景与意义
在储能系统 (ESS) 的开发中，协议栈的稳健性直接影响系统安全。传统的正向测试难以覆盖协议畸形、时序竞争等异常场景。本框架通过模拟“恶意从站”，填补了协议负向测试的空白，能够有效发现缓冲区溢出、状态机死锁等深度隐患。

## 2. 总体架构设计

本框架采用 **数据驱动 (Data-Driven)** 与 **引擎解耦** 的设计思路。

### 2.1 架构组件
*   **仿真引擎 (The Engine)**: `malicious_simulator.py` 与 `vulnerable_target.py`。基于 Python `asyncio` 异步框架实现。其中 `vulnerable_target.py` 采用 **插件化 (Pluggable) 架构**，将漏洞模拟逻辑从主循环中剥离，支持通过 Hook 机制动态注入故障。
*   **漏洞插件 (The Plugins)**: 位于 `vulnerabilities/` 目录，涵盖 OOB、LEAK、DELAY 等多种工业通信常见漏洞。
*   **点表插件 (The Profile)**: `*.json` 文件。定义了寄存器空间、初始状态以及蜜罐阈值。
*   **验证套件 (The Test Suite)**: 基于 `pytest` + `pymodbus` 构建的分层测试脚本，涵盖从基础读写到高并发异常的完整生命周期。

### 2.2 核心数据模型 (DataStore)
仿真器内部维护两个内存 Map：
*   `self.registers`: 存储 16-bit 的保持寄存器与输入寄存器。
*   `self.coils`: 存储 1-bit 的线圈与离散输入。
这种设计确保了仿真器具备“有状态”的反馈能力，支持复杂的“写后读”校验逻辑。

## 3. 故障注入算法详解

### 3.1 极值注入 (P4)
*   **原理**: 恶意篡改 MBAP 长度字段或 PDU 数量，使其超过协议栈缓冲区限制。
*   **目的**: 触发受测端的缓冲区溢出或拒绝服务，考察其底层鲁棒性。

### 3.2 蜜罐溢出 (HONEYPOT)
*   **原理**: 动态监测 `0x06` 或 `0x10` 写入指令。当目标地址匹配 Profile 中的蜜罐地址且值超过 `threshold_max` 时，引擎拦截正常写入逻辑，转而返回 `0x80 | FunctionCode` 引导的异常响应码。

### 3.3 事务混淆 (MISMATCH)
*   **原理**: 捕获请求的 `Transaction ID (TID)`，在生成响应报文时，对其进行非线性偏移（如 `(TID + 999) % 65536`）。
*   **目的**: 测试受测程序是否对 TID 进行强一致性匹配。

### 3.4 慢响应叠加与资源耗尽 (SOAK / STRESS)
*   **模拟场景**: 针对真实的 BMS/PCS 隐患（如响应时间 5s，查询周期 2s）。
*   **原理**: 使用 `test_modbus_soak.py` 发起非阻塞异步请求，当 `Interval < Response Time` 时，受测端会产生挂起的 TCP 连接堆积。
*   **监控重点**: 观察受测端的 Socket 句柄释放速度、内存波动（是否由于请求上下文未释放导致 OOM）以及 P1 维度的业务心跳是否会被 P3/P5 压力压垮。

## 4. 关键技术实现

### 4.1 异步防死锁机制
在异步 IO 读取中，采用了 `asyncio.wait_for(reader.readexactly(N), timeout=5.0)`。
*   **设计初衷**: 防止恶意客户端（或因故障注入导致的异常状态）长时间占用 Socket 资源而不发送数据，确保仿真引擎本身的鲁棒性。

### 4.2 工业级协议支持
*   **多寄存器写入 (0x10)**: 实现了复杂的 PDU 拆解逻辑（起始地址读取、寄存器个数校准、字节序列转换），使其能支持真实的工业控制场景（如批量设置 PCS 参数）。

## 5. 收益与推广手册
*   **复用性**: 其他团队只需根据自己的设备点表编写 JSON Profile，即可复用所有的异常模式。
*   **自动化**: 配合 `run_all_tests.sh` 可轻松集成进 CI/CD 流水线。
*   **可视化**: 自动集成抓包功能，故障发生时直接提供 `.pcap` 证据文件，无需手动抓包。
